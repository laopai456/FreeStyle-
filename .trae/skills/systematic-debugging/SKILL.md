# 系统化调试（逆向工程适配版）

## 铁律

```
不做根因调查，不许提修复方案
不做根因调查，不许提修复方案
不做根因调查，不许提修复方案
```

## 何时使用

| 场景 | 示例 |
|------|------|
| 补丁后崩溃 | Code Cave 注入后切换频道崩溃 |
| 内存操作失败 | WriteProcessMemory 返回错误 |
| 找不到目标对象 | 堆上扫描不到 DStaticActor |
| APOLLO 杀进程 | 代码段 CRC 检测触发 |
| 游戏异常行为 | 发型显示为光头、物理效果缺失 |
| 反复尝试多种方案 | 方案A/B/C/D 都失败 |

## 四个阶段

### 第一阶段：根因调查

**在尝试任何修复之前必须完成：**

1. **仔细阅读错误信息**
   - 游戏崩溃日志、Windows 事件日志
   - Python 脚本的异常输出
   - x64dbg 的异常地址和错误码

2. **稳定复现**
   - 崩溃是否每次都在同一地址？
   - 是否只在特定操作后崩溃（切换频道、装备道具）？
   - 不打补丁时是否正常？

3. **对比分析**
   - 正常流程（DDynamicActor）vs 异常流程（被 patch 的 DStaticActor）
   - 对比基类构造函数差异
   - 对比内存布局差异

4. **在内存中收集证据**
   - 读崩溃地址附近的内存
   - 检查 vtable 指针是否正确
   - 检查 flags 和 motion type 字段
   - 检查物理参数是否为非零

### 第二阶段：模式分析

1. **找到正常工作的参考**
   - DDynamicActor 构造流程（0x0229B0B0）
   - DynamicInit 初始化流程（0x0229B4D0）
   - 基类构造（0x02299BD0）

2. **与异常情况对比**
   - DStaticActor 构造流程（0x0236B8A0）
   - 基类构造（0x0236B9F0）
   - 列出每一个差异

3. **识别关键差异**
   - 内存布局差异（偏移、字段大小）
   - 初始化顺序差异
   - 前置条件差异

### 第三阶段：假设与验证

1. **提出单一假设**
   - "我认为崩溃是因为 DStaticActor 的基类构造没有初始化偏移 0x460 的物理参数"
   - 要具体，不要含糊

2. **最小化验证**
   - 只改一个变量
   - 用 x32dbg 在崩溃地址设断点
   - 读寄存器和内存验证假设

3. **继续之前先验证**
   - 生效了 → 进入第四阶段
   - 没生效 → 提出新假设，不要叠加修复

### 第四阶段：实施

1. **实施单一修复**
   - 只修复已定位的根本原因
   - 不做"顺便改改"

2. **验证修复**
   - 游戏是否正常运行？
   - 物理效果是否出现？
   - APOLLO 是否检测到？

3. **如果修复不起作用**
   - 少于 3 次：回到第一阶段
   - **3 次或以上：停下来质疑方案**

## 红线

| 想法 | 正确做法 |
|------|---------|
| "先试一下，再排查" | 停下来，回到第一阶段 |
| "大概是这个地址的问题" | 用数据验证，不要猜 |
| "再试一次修复" | 如果已经 3 次了，质疑方案 |
| "同时改多个地方" | 每次只改一个变量 |
| "跳过验证，直接测" | 没有验证证据不许宣称完成 |

## 3 次失败后的架构审视

当你已经尝试了 3 种以上补丁方案都失败时：

1. **质疑根本假设**
   - "头发 actor 真的是 DStaticActor 吗？"
   - "vtable 替换方案从根本上可行吗？"
   - "物理效果真的是由对象类型决定的吗？"

2. **考虑替代方案**
   - 方案 D：Hook 工厂函数
   - 方案 E：Hook CharacterMotion 解析
   - 方案 F：内核驱动绕过

3. **在尝试更多修复之前，和用户讨论**
