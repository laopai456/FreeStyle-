# 动态发型逆向调试总结

## 一、问题定义

游戏有两种发型：
- **静态发型**：纯模型，直接替换 BML 文件即可换装
- **动态发型**：带物理骨骼（Bone01-Bone11），有摆动效果

**目标**：让静态发型角色使用动态发型（带物理效果）

**现状**：直接替换 BML 文件后角色变光头

---

## 二、文件格式发现

### 2.1 BML 文件
- 存储在 `itemXXX.pak` 中
- 内容是 XOR 0xFF 加密的 XML
- 包含 `<mesh>` 路径（指向 res pak 中的 SMD 文件）和 `<texture>` 路径（指向 PNG）
- 示例路径：`res764\i50125031_MT.smd`

### 2.2 SMD 文件（SSKF 格式）
- 存储在 `resXXX.pak` 中
- 魔数：`SSKF`，版本 `01 00 00 00`
- **静态发型 vs 动态发型的关键差异**：
  - 静态发型：只有标准 Bip01 骨骼（~44个），文件较大（~919KB）
  - 动态发型：额外包含 Bone01-Bone11 物理骨骼（共~55个），文件较小（~353KB）

### 2.3 动态发型独有骨骼
```
Bone01, Bone02, Bone03(mirrored), Bone03(mirrored)(mirrored),
Bone03, Bone03(mirrored)234, Bone04, Bone05, Bone06,
Bone06(mirrored), Bone07, Bone08, Bone09, Bone09(mirrored),
Bone10, Bone11
```

### 2.4 PGFN Pak 格式
- 魔数：`PGFN`
- item pak 和 res pak 都是此格式
- **不是简单的文件名+偏移索引**
- 每个文件块结构：`[16字节头部][null结尾文件名][加密数据]`
  - 头部[+0x00]：文件名长度（如 0x0D = 13）
  - 头部[+0x04]：数据起始偏移
  - 头部[+0x08]：数据大小
  - 头部[+0x0C]：下一块偏移
- 块之间通过 next_block 指针链接
- BML 数据是 XOR 0xFF 加密

## 四、已验证的替换尝试

### 4.1 Python 重打包 item767.pak ❌（灾难性失败）
- **做法**：用 Python 自写打包脚本替换 BML 后重打整个 pak
- **结果**：pak 格式完全搞坏，item767 所有服装消失
- **教训**：PGFN 格式不是简单格式，不能自己猜着写打包逻辑

### 4.2 PGFN 精确替换 BML 数据（大小不同） ❌（游戏崩溃）
- **做法**：解析 PGFN 块结构，替换 BML 数据，更新后续偏移
- **结果**：游戏无响应
- **教训**：偏移更新逻辑可能有遗漏，PGFN 内部结构比想象的复杂

### 4.3 PGFN 精确替换 BML 数据（大小相同） ❌（光头）
- **做法**：补齐空格使新 BML 大小与原始一致，不需要改偏移
- **结果**：游戏不崩溃但发型不显示（光头）
- **结论**：BML 替换成功，pak 结构完好，但游戏加载动态发型 SMD 失败

### 4.4 当前结论
**BML 替换方案行不通**——游戏能读到动态发型的 BML，但加载对应的 SMD 文件时失败了。原因未知，需要调试确认。

### 9.4 运动类型（Motion Type）系统

#### FName 系统
游戏使用类似 Unreal Engine 的 FName 系统，通过整数索引引用字符串：

| FName 名称 | 全局对象地址 | 索引值 |
|------------|-------------|--------|
| "Base" | 0x2A783F8 | 10 (0x0A) |
| "Dynamic" | 0x2A78410 | 11 (0x0B) |
| "Static" | 0x2A783F4 | 12 (0x0C) |

FName 注册函数：
- Base: 0x26892A0
- Dynamic: 0x26892D0
- Static: 0x2689300
- 底层解析器: 0x021D8C40

#### 运动类型存储
- **对象偏移 0xC0**：存储运动类型的 FName 索引
- Dynamic 对象 → 0xC0 = 11 (0x0B)
- Static 对象 → 0xC0 = 12 (0x0C)

#### 关键函数

| 函数 | 地址 | 功能 |
|------|------|------|
| DynamicInit | **0x229B4D0** | 设置 Dynamic + BaseInit + PhysicsInit |
| StaticInit | **0x24C5910** | 仅设置 Static + BaseInit，无物理 |
| SetMotionType | **0x02297810** | 写入 this+0xC0 的运动类型 |
| BaseInit | **0x02295B10** | 基础初始化 |
| DynamicPhysicsInit | **0x0229C2D0** | Dynamic 专属物理初始化 |
| BML ItemCode 解析器 | **0x21B6070** | 解析 "ItemCode%d" 格式 |

#### DynamicInit 伪代码
```cpp

    BaseInit(this);                          // call 0x02295B10

```

#### StaticInit 伪代码
```cpp
void DStaticActor::StaticInit(this) {
```

### 9.5 物理初始化条件

函数 `0x0229C2D0` (DynamicPhysicsInit) 的关键检查：

```cpp
void DynamicPhysicsInit(this, float param1, float param2) {
    if ((this->flags_0x74 & 0x18) == 0) return;   // 必须启用物理
    if (this->field_0x460 == 0) return;             // 物理参数必须非零
    // ... 物理初始化 ...
    if (this->flags_0x74 & 0x10000) {
        // 物理模式 A
    } else {
        // 物理模式 B
    }
}
```

**关键偏移**：
| 偏移 | 类型 | 用途 |
|------|------|------|
| 0x74 | DWORD | 控制标志：& 0x18 启用物理，& 0x10000 物理模式 |
| 0xC0 | DWORD | 运动类型 FName 索引 |
| 0x460 | float | 物理参数1（使用前取负） |
| 0x464 | float | 物理参数2 |
| 0x468 | float | 物理参数3 |
| 0x46C | DWORD | 传递给 vtable[0xA0] 的参数 |

### 9.11 CharacterMotion.bml 解析流程

#### BML 字段结构

CharacterMotion.bml 中每个角色条目包含以下字段：

| 字段名 | 地址 | 说明 |
|--------|------|------|
| ItemCode%d | 0x283C76C | 道具代码（循环索引） |
| ItemCount | 0x283C778 | 道具总数 |
| CharacterName | 0x283C788 | 角色名称 |
| Face | 0x283C704 | 脸型 |
| **Type** | 0x26A920C | **运动类型（关键！）** |
| **MotionName** | 0x283C6F8 | **运动名称（关键！）** |
| MotionRotate | 0x283C6E8 | 运动旋转 |
| MovieName | 0x283C6B8 | 动画名称 |
| Effect_1 | 0x283C6DC | 效果1 |
| Effect_2 | 0x283C6D0 | 效果2 |
| Effect_3 | 0x283C6C4 | 效果3 |

#### BML 路径

BML 数据路径：`./CharacterMotionList/Character`（Unicode，地址 0x283C710）

#### 解析函数

| 函数 | 地址 | 功能 |
|------|------|------|
| CharacterMotion 解析主函数 | **0x21B46E0** | 解析 CharacterMotion.bml，为每个道具创建运动对象 |
| BML ItemCode 循环 | 0x21B6070 | 遍历 ItemCode%d 格式字段 |
| 对象创建工厂 | **0x021C1F00** | 根据 vtable 指针创建运动对象 |
| vtable 复制函数 | 0x021C13E0 | 从 Type 值获取 vtable 指针并复制 |
| 树查找/插入 | 0x021BD690 | 管理运动对象映射表 |

#### Type 字段决策流程

```
CharacterMotion.bml (服务器下发)
  → 解析每个 ItemCode 条目
  → 读取 Type 字段值
  → Type 值通过虚函数调用转换为 vtable 指针
  → 0x021C1F00 根据 vtable 创建 DDynamicActor 或 DStaticActor
  → DDynamicActor: vtable=0x284AA4C, 包含物理初始化
  → DStaticActor:  vtable=0x284E114, 无物理初始化
```

#### 关键发现：CharacterMotion.bml 不在本地

搜索了游戏目录所有 PAK 文件（包括 ur_event_MyMotionPick.pak、ur_event_MotionChange.pak、ur_game.pak 等），**CharacterMotion.bml 在明文和 XOR 0xFF 加密两种编码下均未找到**。

结论：**CharacterMotion.bml 由服务器动态下发**，不在本地文件系统中。这意味着：
- 无法通过修改本地 BML 文件来改变运动类型
- 运动类型由服务器配置决定
- 修改方案必须是运行时的

## 十、修改方案分析

### 方案 B-1：BML 配置修改 ❌ 不可行
- CharacterMotion.bml 由服务器下发，不在本地
- 即使拦截服务器数据包修改，也可能被校验

### 方案 B-2：运行时内存补丁 ⚠️ 可行但需解决 APOLLO
- 修改对象偏移 0xC0 的运动类型字段（0x0C→0x0B）
- 修改对象偏移 0x74 的物理标志（设置 & 0x18）
- 修改对象偏移 0x460 的物理参数（设为非零）
- 替换 vtable 指针（0x284E114→0x284AA4C）
- **问题**：需要 WriteProcessMemory，APOLLO 可能监控

### 方案 C：DLL 注入 Hook ✅ 最可行
- Hook DStaticActor 构造函数 (0x236B8A0)
- 在构造完成后，替换 vtable 为 DDynamicActor 的 vtable
- 调用 DynamicInit (0x229B4D0) 重新初始化
- **优势**：只需 Hook 一个函数，改动最小
- **风险**：APOLLO 可能检测 DLL 注入

### 方案 C-2：DLL 注入 + SetMotionType 调用 ✅ 推荐
- Hook CharacterMotion 解析函数 (0x21B46E0)
- 在解析完成后，找到目标道具的运动对象
- 调用 SetMotionType (0x02297810) 将类型改为 Dynamic
- 替换 vtable 指针
- 调用 DynamicPhysicsInit (0x0229C2D0) 启用物理
- **优势**：在正确的时机修改，确保物理系统正确初始化

### 方案 C-3：网络数据包修改 ⚠️ 复杂
- 拦截服务器下发的 CharacterMotion.bml 数据
- 修改 Type 字段值
- **问题**：需要理解网络协议，可能被校验

## 十一、补丁实测记录

### 11.1 第一次实测（2026-05-14）

**方案**：Python WriteProcessMemory + Code Cave
**工具**：`dynamic_hair_patch.py`
**补丁位置**：DStaticActor 构造函数 0x236B998
**补丁内容**：
- `this+0x74 |= 0x18`（启用物理标志）
- `this+0x460 = 1.0f`（物理参数，如果为零）
- `this+0x00 = 0x284AA4C`（替换 vtable 为 DDynamicActor）
- `this+0xC0 = 0`（清除运动类型）
- `call DynamicInit (0x0229B4D0)`（完整动态初始化）

**Code Cave 地址**：0x033B0000（76字节）
**补丁结果**：写入成功，验证通过

**实测结果**：❌ **切换频道时游戏失去响应（崩溃）**

**崩溃分析**：
补丁在 DStaticActor 构造函数末尾替换了 vtable 和 init 调用。切换频道触发角色重新加载时，构造函数被调用，补丁代码执行，但可能导致以下问题之一：
1. DynamicInit 内部调用的函数假设某些前置条件已满足（如基类构造差异）
2. DStaticActor 和 DDynamicActor 的基类构造不同（0x0236B9F0 vs 0x02299BD0），内存布局可能有差异
3. DynamicInit 中调用 vtable[0xA0] 时使用了错误的参数（0x46C 偏移）
4. 栈帧（SEH 状态）不匹配导致异常处理失败

**改进方向**：
- [ ] 用 x32dbg 在 code cave 设断点，单步执行定位具体崩溃指令
- [ ] 对比 DStaticActor 和 DDynamicActor 完整构造流程的差异
- [ ] 尝试更保守的补丁：只改 vtable 不改 init（让运行时代码通过 vtable 路由到 DDynamicActor 方法）
- [ ] 尝试在对象创建后（而非构造中）修改属性（Hook BML 解析完成后的代码）
- [ ] 检查 DStaticActor 的基类 0x0236B9F0 和 DDynamicActor 的基类 0x02299BD0 的初始化差异
