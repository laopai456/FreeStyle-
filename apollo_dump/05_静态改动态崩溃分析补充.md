# 05_静态改动态崩溃分析补充.md

> 目的：回答当前最实际的问题——**已知静态改动态会崩，下一步不是重复证明这一点，而是要判断“谁在杀、在哪一层杀、真正突破口在哪里”。**
>
> 结论先行：基于目前全部现象，**主崩点更像游戏业务层崩，而不是 Apollo 主动杀**；Apollo 仍然可能干扰，但不是当前“进练习场才崩”的首要解释。

---

## 一、当前问题重述

已知现象链：

- 路径替换成功
- 资源命中成功
- 发型能渲染
- **进入练习场崩溃**

这说明至少下列阶段已经成功通过：

1. hook/替换逻辑生效
2. BML 查找链生效
3. SMD/贴图基础资源可加载
4. 房间/预览阶段的基础渲染成立

因此，当前要分析的不是“资源为什么没命中”，也不是“静态改动态会不会崩”——这个已经知道了。

**真正的问题是：这次崩溃到底属于哪一类？**

- Apollo 主动检测并杀进程
- 游戏在进入练习场阶段走到更深的动画/物理/对象更新链后，因类型不匹配而自己崩

---

## 二、核心判断：先分清“被杀”还是“自己崩”

### 2.1 Apollo 杀的典型特征

通常具备以下特征：

- 注入后很快闪退，甚至未登录即死
- 崩溃地址常落在 **Apollo 模块地址区**
- 崩溃指令常见 `cc`（INT3）
- 与“是否进入练习场”不强绑定，登录前、登录后、任意完整性扫描点都可能触发
- 更像“行为被发现后的处置”而不是“场景逻辑运行失败”

### 2.2 游戏业务崩的典型特征

通常具备以下特征：

- 可以先正常显示、先正常渲染、先正常进前置场景
- 在**进入练习场 / 第二次加载 / 动画更新 / 物理初始化**时稳定复现
- 崩溃地址更常落在 **游戏主模块** 或其调用链附近
- 常见空指针/野指针/对象字段未初始化
- 与“静态对象被喂动态资源”高度相关

### 2.3 对当前案例的初步判定

当前现象是：

- 路径替换成功
- 资源命中成功
- 发型能渲染
- **进入练习场才崩**

按概率判断，更像：

> **不是 Apollo 在看到替换成功后“顺手杀你”，而是游戏在进入练习场阶段，开始真正走动态骨骼/物理/动画链路时，把静态 Actor 当成动态资源消费者使用，最终访问了不存在的状态字段。**

也就是说：

> **当前主崩点更偏向业务崩，不是检测杀。**

---

## 三、为什么我更倾向“业务崩”而不是“Apollo 杀”

### 3.1 资源与初次渲染已经成立

如果是 Apollo 纯完整性处置，常见情况是：

- 注入不久就死
- hook 安装后不久就死
- 登录前/登录后某固定周期死

而你这里的链路是：

- hook 已生效
- BML 已命中
- SMD 已加载
- 发型已渲染
- **只在练习场阶段崩**

这说明：

- Apollo 若要杀，至少不是最早那一层在杀
- 当前崩点与“进入完整 gameplay 初始化”强相关

### 3.2 练习场是“第二阶段逻辑”

房间/预览和练习场不是一回事。

进入练习场后，通常会新增这些操作：

- 角色重新实例化或切换运行态
- 动画系统开始稳定驱动
- 物理骨骼/摆动/碰撞初始化
- 第二次资源加载或角色变体补齐
- motion type / actor type 对应逻辑正式生效

如果静态 Actor 只是“勉强挂上动态资源”，前一阶段可能还能显示；但后一阶段一旦访问动态专属字段，就很容易直接崩。

### 3.3 你的文档已有明确支持证据

在《04_练习场崩溃诊断手册》中，P0 已经是：

- **DStaticActor 被替换为 DDynamicActor 的 SMD**
- 静态 Actor 只有 59 骨，无物理骨骼
- 动态 SMD 有 75 骨，含 16 根物理骨骼
- 练习场动画系统会访问物理骨骼区域
- 最终读到空指针/野指针

这条证据链，本身已经比“Apollo 延迟处置”更贴合当前现象。

## 四、当前真正要回答的问题

不是：

- 静态改动态会不会崩？（已知会）
- 资源是不是命中了？（已知命中）

而是：

### 4.1 崩溃发生在哪一层？

- Apollo 模块内部？
- 游戏主模块角色初始化？
- 动画更新？
- 物理初始化？

### 4.2 最后一个成功点是什么？

- `sprintf` 替换成功？
- `LoadItemFile` 成功？
- 第一次 `AcquireSMD` 成功？
- 第二次 `AcquireSMD` 成功？
- Actor 创建成功？
- 进入动画/物理更新后崩？

### 4.3 崩溃当时对象状态是什么？

关键不是“资源文件长什么样”，而是：

- 这个 Actor 还是不是 `DStaticActor`
- `vtable` 是什么
- `motion type` 是什么
- 物理参数块指针是否存在
- 进入练习场前后这些值有没有变化

## 五、下一步最该做的调试：不是继续试资源，而是“定性崩溃”

### 5.1 必须记录的 4 个信息

每次崩溃都至少记：

1. **崩溃地址**
2. **崩溃模块**
3. **崩溃指令**
4. **崩溃前最后一条自定义日志**

判定规则：

- **Apollo 模块 + `cc`/INT3** → 更像 Apollo 杀
- **游戏模块 + 空指针/对象访问异常** → 更像业务崩

### 5.2 把时间线拆成 4 段

对每次实验都按下面 4 段记录：

1. 注入后，未登录
2. 登录后，大厅/房间
3. 进入练习场加载中
4. 练习场第一帧 / 第二次资源加载 / 动画更新阶段

用途：

- 如果未登录或刚登录就死 → 更像 Apollo
- 如果房间正常、进练习场才死 → 更像业务崩

### 5.3 记录“最后一个成功点”

建议盯住这几个节点：

- `sprintf` 是否替换成功
- `LoadItemFile` 是否拿到目标 ItemCode
- 第一次 `AcquireSMD` 是否成功
- 第二次 `AcquireSMD` 是否成功
- 第二次成功后是否紧接着进入 Actor 更新/动画更新

如果链路是：

- 路径替换成功
- BML 命中
- SMD 成功加载
- 房间渲染成功
- **练习场第二次加载/动画更新时死**

那就更支持：

> **业务崩 > Apollo 杀**

## 六、真正该盯的观察点：对象状态，而不是资源状态

后续不要只盯：

- BML 内容
- 文件路径
- SMD 是否存在

这些在当前阶段已经不是第一矛盾。

### 6.1 更关键的字段

进入练习场前后，重点观察：

- Actor 地址
- `vtable`
- `+0xC0` motion type
- `+0x460` 物理参数块指针

### 6.2 要回答的核心问题

> **房间里显示这个发型的对象，进入练习场时，到底还是不是静态 Actor？它有没有被后续逻辑当成动态对象来更新？**

如果答案是：

- `vtable` 仍然是静态 Actor
- 但后续逻辑开始走动态链路
- 或访问了动态对象才有的物理字段

那么根因就非常清楚：

> **对象创建类型与资源类型不一致，导致运行时访问越界/空指针。**

---

## 七、为什么“继续隐藏 hook”不是当前主突破口

如果最终证据指向业务崩，那么继续把精力放在：

- 更隐蔽的注入
- 更花的 Frida 技巧
- 更复杂的资源替换

都不是当前主突破口。

因为问题已经从“能不能替换成功”转移为：

> **替换成功后，游戏对象生命周期是否仍然合法。**

也就是说，这已经不是单纯“资源层问题”，而是：

- 对象创建链
- 类型分流逻辑
- 动态初始化前置条件
- 动画/物理系统的调用契约

---

## 八、如果目标是“让它不崩”，真正的三类突破口

> 这里只做方向性分析，不给实现细节。

### 8.1 方向 A：对象创建时就与资源类型一致

思路：

- 不让静态对象去消费动态资源
- 而是让该对象从创建开始就走动态类型链路

本质：

- 解决“对象类型 vs 资源类型”错配

### 8.2 方向 B：让静态对象补齐动态链路所需的最小状态

思路：

- 如果仍想保留静态对象壳子，就必须补齐动态更新路径需要的字段/状态

风险：

- 这通常不是补 1~2 个字段就能完事
- 很可能涉及构造顺序、基类初始化、物理参数块、motion type 等一整串前置条件

### 8.3 方向 C：阻断后续逻辑访问动态专属链

思路：

- 不去“变成动态对象”，而是让后续系统不要进入那些必须依赖动态物理/骨骼的路径

限制：

- 这条路通常会损失动态效果
- 但作为验证路线，有时比“完整补齐动态对象”更容易先拿到稳定性结论

## 九、当前阶段最合理的结论

综合现有现象，我的判断是：

### 9.1 这次崩溃的首要嫌疑

**游戏业务崩**，不是 Apollo 首杀。

### 9.2 最可能的具体触发点

- 进入练习场后
- 第二次资源/角色阶段加载
- 动画或物理更新开始
- 静态 Actor 被动态资源链路当成合法动态对象使用
- 访问了不存在的物理骨骼/状态块/初始化字段

### 9.3 当前调试重心应该转到哪里

从“资源命中没命中”转到：

- 崩溃模块归属
- 最后成功点
- Actor 对象状态
- 进入练习场前后类型/字段变化

## 十一、最新进展：练习场 ItemCode 覆盖问题（2026-05-31）

### 11.1 问题描述

在 `[ebp-0x478c]+0x164+0x7DC` 这个字段上做了 practice hook，每次进练习场都将 ItemCode 修复为 `50125711`。但：

- **发型仍然没有显示**（恢复默认）
- 说明 `+0x7DC` 字段可能不是唯一的决定因素

### 11.2 关键日志证据

```
23:56:03.604 - practice hook: itemCode=0 -> FIXED to 50125711
23:56:27.087 - practice hook: itemCode=0 -> FIXED to 50125711
```

两个关键观察：

| 观察 | 含义 |
|:---|:---|
| **itemCode=0** | 练习场将发型 ItemCode **清零了**（不是覆盖为其他值，而是设为0） |
| **sprintf #8,#18,#28,#38,#48 全是 50125461** | sprintf 输出缓冲区里全是原始 ItemCode，50125711 从未出现在 sprintf 路径中 |

### 11.3 数据流完整路径（更新版）

```
[角色对象] ──+0x7DC──→ ItemCode 缓存 (在练习场被清零为 0)
     │
     ├── sprintf 路径 ──→ "customize\item\i%d.xml" ──→ LoadItemFile
     │      ↑ 这里走的还是 50125461，不是 50125711
     │
     └── 装备链表路径 ──→ node+0x10 ──→ sprintf 参数
            ↑ 链表内容可能与 +0x7DC 不同步
```

### 11.4 三个可能的原因

**原因 1：修复时机太晚**

`+0x7DC` 的值在 practice hook 触发前就已经被用于加载了。hook 在 `0x173C601`（读 this 指针）处触发，但发型加载可能在更上游的 `0x1AE1F80` 函数入口之前就已经完成了。

证据：sprintf 输出里全是 `50125461`，说明 LoadItemFile 链使用的是原始 ItemCode，不是修复后的。

**原因 2：需要同时修复多个字段**

过去在 sprint#1 的运行时 dump 中，扫描到 **11 个对象** 包含 `50125461`，包括：
- `0x193AB0+0x0` — 可能是装备槽结构第一个字段
- `0x1939BC+0xF4` — 角色装备数组
- `0x193A1C+0x94` — 装备结构
- `0x6A17F4D8+0x10` — DLL 数据区缓存
- `0x6A17F410+0xD8` — DLL 数据区缓存

这些字段分布在角色对象、装备数组、DLL 缓存三个层级。只修复 `+0x7DC` 一个点可能不够。

**原因 3：发型渲染不依赖这个字段**

`+0x7DC` 可能只是"存储/缓存"字段，实际渲染使用的数据来源是：
- 装备链表 (`node+0x10`) — sprint 的 ItemCode 参数来源
- 或其他未发现的渲染专用字段

### 11.5 下一步方向

| 优先级 | 方向 | 做法 |
|:---:|:---|:---|
| P0 | **在 sprintf 调用点修复** | 继续用原 `hook_sprintf_replace.py` 方案（已经验证在房间/练习场 sprintf 输出中都替换成功） |
| P1 | **同时修复链表 + 缓存** | hook `0x1AE1F80` 入口，同时修复 `[ebp+0x14]` 链表的 node+0x10 和 `[ebp-0x3C0]+0x7DC` |
| P2 | **跟踪清零源头** | 找到谁把 `+0x7DC` 写成 0，确认清零发生在哪个阶段 |

## 十二、最终一句话结论

> **当前真正要做的，不是继续证明"静态改动态会崩"，而是把这次崩溃先定性为 Apollo 杀还是业务崩。按现有证据，更大概率是：游戏在练习场阶段走到动态骨骼/物理更新链时，把静态对象当成动态对象用了，因此发生业务层崩溃。**
