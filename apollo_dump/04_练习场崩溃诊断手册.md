# 04_练习场崩溃诊断手册 — 静态改动态崩溃根因排查

> 基于 2026-05-18 ~ 05-31 全部实测数据整理。
> 按 **骨骼 → 物理 → LOD → 配置表 → 校验** 五条线展开。
>
> 标记：★ = 最容易导致「进入练习场直接崩溃」

---

## 零、速查：先看哪个？

| 优先级 | 线 | 一句话 | 如果 |
|--------|-----|--------|------|
| **P0** | 骨骼 | Actor 类型不匹配 → 动画系统访问不存在的物理骨骼 | ★ 进练习场秒崩 |
| **P1** | 配置表 | BML 路径/Group 错误 → 文件找不到 → 光头 | 进房间光头 |
| **P2** | 校验 | Apollo 检测到 hook → TerminateProcess | 几秒后被杀 |
| **P2** | 物理 | 物理参数未初始化 → DynamicPhysicsInit 返回 → 无效果 | 渲染正常但无摆动 |
| **P3** | LOD | BML 引用不存在的 mesh 变体 → 加载失败 | 特定角色不显示 |

## 一、骨骼（Skeleton）

### 1.1 常见触发原因

| # | 原因 | 严重度 |
|---|------|--------|
| **★** | **DStaticActor 被替换为 DDynamicActor 的 SMD**：静态 Actor 只有 59 根骨骼（无物理骨骼），而动态 SMD 有 75 根骨骼（含 16 根物理骨骼）。练习场动画系统通过 vtable 调用动画更新，访问 +0x460 区域的物理骨骼数据时读到空指针/野指针 | 🔴 必崩 |
| ★ | **跨 Actor 类型替换**：sprintf hook 把静态发型(50125461)的 BML 指向动态发型(50125711)，但 Actor 创建时仍按静态类型构造 → vtable 指向 DStaticActor → 虚函数表不包含物理更新入口 | 🔴 必崩 |
| | BML 引用不存在的 SMD 后缀（如 `_MT.smd` 实际只有 `_FN.smd`） | 🟡 可能崩 |
| | SSKF 格式解析错误：SName 实际是 null-term ASCII（非长度前缀），Scale 字段在 v1 中存在 | 🟡 解析错乱 |

### 1.2 优先检查项

1. **源发型和目标发型的 Actor 类型是否一致**：都静态？都动态？还是跨类型？
2. **目标发型的 res PAK 中实际存在哪些 SMD**：`py analyze_super_saiyan2.py`
3. **BML 中引用的 mesh 路径是否全部存在于 res PAK**
4. **SSKF 解析是否正确**：用 `debug_sskf.py` 做 round-trip 验证

### 1.3 对应日志/报错特征

```
[AcquireSMD] #10 path="res768\i50125711_FN.smd"  ← 静态→动态替换，第一次加载正常
[ReadFile] #20 name="SSKF" bytes=269824           ← 大mesh加载成功
...
[替换!] #2  (第二次加载 — 练习场)                 ← 第二次触发
[AcquireSMD] #22 path="res768\i50125711_FN.smd"   ← 再次加载同一个文件
...
[ReadFile] #164 name="SSKF" bytes=34816           ← 崩溃！(ReadFile #164)
```

崩溃地址特征：
- EIP 在 `0x69000000` 区域（Apollo 模块）或 `0x01xxxxxx` 区域（游戏代码）
- 崩溃指令含 `cc`（INT3 断点）— Apollo 的检测注入
- EAX=0x0（空指针解引用）— 物理骨骼数据缺失

### 1.4 最快排除方法

```powershell
# 1. 确认目标发型是静态还是动态
py analyze_super_saiyan2.py
# 看输出中 subtype_058:
#   0=静态, 1=静态, 2=动态, 3=静态, 4=动态, 5=动态

# 2. 只替换同类型发型（静态→静态，动态→动态）
# 修改 hook_sprintf_replace.py 中的 DST_IC 为同类型 ItemCode

# 3. 如果必须跨类型，需要同时 Hook Actor 工厂函数
```

### 1.5 如果命中，通常怎么修

**方案 A（推荐）**：只替换同类型 Actor 的发型
```python
# hook_sprintf_replace.py
SRC_IC = 50125461  # 美丽梦想（静态）
DST_IC = 50125691  # 黑金热血青春（静态）— 同类型，不会崩
```

**方案 B（复杂）**：Hook Actor 工厂强制创建 DDynamicActor
- Hook `0x021C1F00`（工厂函数），根据 ItemCode 决定创建 DStaticActor 还是 DDynamicActor
- 或 Hook `SetMotionType(0x02297810)` 强制写入 `FName("Dynamic")` = 0x0B

**方案 C（最复杂）**：Hook 动画系统，patch 掉不存在的物理骨骼引用
- 风险极高，需要完整理解动画管线

### 关键数据速查

| 常量 | 值 | 说明 |
|------|-----|------|
| DStaticActor vtable | 0x0284E114 | 59 骨，无物理 |
| DDynamicActor vtable | 0x0284AA4C | 75 骨，16 物理骨骼 |
| 物理骨骼数 | 16 | Bip01 Head 有 2 个子节点 |
| DStaticActor ctor | 0x236B8A0 | 构造入口 |
| DDynamicActor ctor | 0x229B0B0 | 构造入口 |
| 工厂函数 | 0x021C1F00 | Type→对象创建 |

## 二、物理（Physics）

### 2.1 常见触发原因

| # | 原因 | 严重度 |
|---|------|--------|
| | DStaticActor 被 patch 为 DDynamicActor vtable，但物理参数未初始化（+0x460=0, +0x74 flags 不对） | 🟡 无效果 |
| ★ | DynamicInit (0x229B4D0) 调用 `vtable[0xA0](this, this->field_46C)`，但基类构造不同导致 SEH 异常链断裂 | 🔴 必崩 |
| | Code Cave 补丁位置错误：在构造函数中间修改 vtable → 后续初始化访问错误偏移 | 🔴 必崩 |
| | 全局物理参数指针 `[0x02A43CA4]` 为 NULL → DynamicPhysicsInit 检查失败 | 🟡 无效果 |

### 2.2 优先检查项

1. **Actor 对象偏移检查**：
   - `+0x00`: vtable 是否正确设置
   - `+0x74`: flags，`& 0x18` 必须非零（启用物理）
   - `+0xC0`: motion type FName 索引（11=Dynamic, 12=Static）
   - `+0x460`: 物理参数块指针，必须非零
   - `+0x46C`: 物理状态，传给 vtable[0xA0]

2. **全局物理参数是否有效**：
   ```python
   # 读取 [0x02A43CA4] → phys_block_ptr
   # 读取 [0x026A1B04] → phys_float
   ```

3. **是否在正确的时机修改**：Actor 完全构造后，而非构造函数中途

### 2.3 对应日志/报错特征

```
[Deep Crash] 地址: 0x694f3e32 (Apollo 模块内)
崩溃指令: cc 9d 17 4e 17 e2 c7 c0...  ← INT3 断点被触发
EAX=0x0 ECX=0x3869764f  ← 垃圾值，动态初始化失败
```

或：
```
[DynamicCreate] eax=0x198a70           ← 创建成功
[DynamicCreate返回] ret=0x86429c28     ← 返回地址异常（负值）
```

### 2.4 最快排除方法

```powershell
# 1. 用 scan_actor_vtable.py 扫描当前 Actor 状态
py scan_actor_vtable.py --dump 0x<addr>   # 查看完整字段

# 2. 对比正常 DDynamicActor 对象
# 检查 +0x74, +0xC0, +0x460 是否与工厂函数设定一致

# 3. 测试：只改 vtable 不改 init → 看是否崩
# 如果崩 → 基类构造差异导致
# 如果不崩但无效果 → 物理参数未初始化
```

### 2.5 如果命中，通常怎么修

**Code Cave 补丁已被证实失败**（§11.1 实测），崩溃原因：
- DynamicInit 内部调用的函数假设基类已正确构造
- DStaticActor 和 DDynamicActor 的基类构造不同（0x0236B9F0 vs 0x02299BD0）
- SEH 异常链在补丁改写后失效

**推荐替代方案**：
- 不修改构造函数，改为 Hook 更上层的工厂函数 (0x021C1F00)
- 或 Hook CharacterMotion 解析函数 (0x21B46E0)，在数据层修改 Type 值

### 物理相关关键字段

| 偏移 | 类型 | 含义 | 静态值 | 动态值 |
|------|------|------|--------|--------|
| +0x00 | DWORD | vtable | 0x284E114 | 0x284AA4C |
| +0x08 | DWORD | 类型标记 | 3 | 1 |
| +0x74 | DWORD | 控制标志 | — | & 0x18 启用物理 |
| +0xC0 | DWORD | 运动类型 FName | 12 (Static) | 11 (Dynamic) |
| +0x460 | DWORD | 物理参数块指针 | 0 | globals(2A43CA4) |
| +0x46C | DWORD | 物理状态 | — | 0 |
| +0x470 | float | 物理参数 | — | float(26A1B04) |
| +0x47C | vec3 | 物理位置 | — | 0,0,0 |
| +0x48C | vec3 | 物理速度 | — | 0,0,0 |

## 三、LOD（Level of Detail / 角色变体）

### 3.1 常见触发原因

| # | 原因 | 严重度 |
|---|------|--------|
| | BML 中 `<character type="N">` 引用的 mesh 变体在 res PAK 中不存在 | 🟡 特定角色光头 |
| | 紫色超赛有 8 个 SMD 变体但 res768.pak 中只有 `_FN.smd` | 🟡 某些角色不显示 |
| | 纹理引用错误（如引用 `i50125701_f.png` 而非 `i50125711.png`） | 🟡 显示异常 |
| | BML 中 `<object type="2">` 残留引用不存在的 MS/FS 变体 | 🟡 解析失败 |

### 3.2 优先检查项

1. **BML 中所有 mesh 引用 vs res PAK 实际内容**：
   ```powershell
   py analyze_super_saiyan2.py   # 列出所有 SMD 和 PNG
   ```

2. **BML 中每个 `<character type>` 的 `<object>` 数量**：
   ```powershell
   py check_orig_bml.py          # 显示原始 BML 结构
   ```

3. **纹理路径是否正确**：BML 中的 PNG 引用是否指向实际存在的文件

### 3.3 对应日志/报错特征

```
[AcquireSMD] #15 path="res768\i50125711_MT.smd"  ← 这个文件在 res768.pak 中不存在！
→ 加载失败 → 该 character type 对应的角色变体显示为光头
```

验证：

```powershell
py -c "
```

### 3.4 最快排除方法


### 3.5 如果命中，通常怎么修

**修正 BML 的 mesh 路径**（[deploy_clean_bml.py](file:///d:/py/反编译/FreeStyle/apollo_dump/deploy_clean_bml.py)）：

```python
replacements = {
    'res768\i50125711_MT.smd': 'res768\i50125711_fn.smd',
    'res768\i50125711_MS.smd': 'res768\i50125711_fn.smd',
    'res768\i50125711_MN.smd': 'res768\i50125711_fn.smd',
    'res768\i50125711_MF.smd': 'res768\i50125711_fn.smd',
```

- XOR 0xFF 编码（LoadBinaryXML 总是做 XOR 解码）

## 四、配置表（Config Tables / BML 查找链）

### 4.1 常见触发原因

| # | 原因 | 严重度 |
|---|------|--------|
| ★ | **BML 的 Group 路径错误**：源码说 Group="item"，实际 binary 用 Group="item768"（根据 ItemCode 动态计算） | 🔴 光头 |
| ★ | **BML 放在错误的松散目录**：`Resource\item\` vs `Resource\item768\` | 🔴 光头 |

| | item PAK 不在搜索路径中 → BML 查找失败 | 🔴 光头 |

### 4.2 优先检查项

1. **BML 文件是否放在正确目录**：

## 五、校验（Apollo Verification）

### 5.1 常见触发原因

| # | 原因 | 严重度 |
|---|------|--------|
| ★ | **ApolloProtect 未完全停止**：`sc stop` 后卡在 STOP_PENDING → Frida 注入时检测到 | 🔴 杀进程 |
| ★ | **Frida Interceptor.attach 在游戏 .text 段**：页面属性 RX→RWX 被 Apollo 检测 | 🔴 杀进程 |
| | ApolloCT.dll 磁盘 CRC 校验 → 客户端版本不符合 | 🔴 拒连 |
| | Apollo.sys ObRegisterCallbacks → 外部 OpenProcess 被封锁 | 🟡 无法注入 |
| | 硬件断点 DR0-DR3 被清零 | 🟡 调试器失效 |
| | Apollo 定时扫描 .text 段页属性 | 🔴 杀进程 |

### 5.2 优先检查项

1. **ApolloProtect 服务状态**：
   ```powershell
   sc.exe query ApolloProtect | Select-String "STATE"
   # 必须是 STATE: 1 STOPPED
   # 如果 STOP_PENDING: 需要重启电脑
   ```

2. **Hook 的目标函数是否在系统 DLL**：
   - ✅ MSVCR100.dll 的 sprintf → 安全
   - ✅ kernel32.dll 的 ReadFile/CreateFileA → 安全
   - ❌ 游戏 .text 段（如 AcquireSMD 调用者 0x01EEBA30）→ 危险

3. **Frida 版本兼容性**：Memory.scan API 在某些版本损坏

### 5.3 对应日志/报错特征

**Apollo 杀进程（TerminateProcess）**：
```
游戏突然消失，无任何错误提示
或：access-violation 在 ~10 秒后
```

**Apollo 检测到 hook（崩溃日志）**：
```
崩溃指令: cc 9d 23 52 e8...  ← INT3 (cc) 是 Apollo 的检测断点
崩溃地址: 0x69557e4d (Apollo 模块内)
```

**Frida 注入失败**：
```
Failed to attach: permission denied
或：进程句柄获取失败
```

### 5.4 最快排除方法

```powershell
# 1. 彻底停 ApolloProtect
sc.exe config ApolloProtect start= disabled
sc.exe stop ApolloProtect
# 如果 STOP_PENDING → 重启电脑

# 2. 只 hook 系统 DLL（不 hook 游戏 .text）
# 主 hook: MSVCR100.dll!sprintf ✅
# 诊断 hook: kernel32.dll!ReadFile ✅
# 危险 hook: 0x01EEBA30 (游戏代码) ❌ — 仅诊断时用

# 3. 使用 rwx→attach→立即恢复 rx 策略（仅冷函数）
# 见 failed_approaches.md §3.7
```

### 5.5 如果命中，通常怎么修

**方案 A（推荐）**：彻底停 ApolloProtect + 只 hook 系统 DLL
```powershell
sc.exe config ApolloProtect start= disabled
# 重启电脑
sc.exe stop ApolloProtect   # 现在应该是 STOPPED
# 启动游戏，运行 hook 脚本
```

**方案 B**：debug_inject.py（调试器注入）
- 以调试器身份启动 FreeStyle.exe
- 在 Apollo DLL 加载瞬间 patch 其 CRC 函数
- 原理：CreateProcess 直接出句柄，绕过 ObRegisterCallbacks

**方案 C**：VirtualQuery 欺骗 + CRC patch
- 原理：hook VirtualQuery 返回假页面属性，同时 patch ApolloCT 的 CRC32 校验函数
- 状态：可行但未完全验证

## 七、当前可用的安全方案

### 方案 1：同类型静态→静态替换 ✅ 最安全

```python
# hook_sprintf_replace.py
SRC_IC = 50125461  # 美丽梦想（静态，subtype=0）
DST_IC = 50125691  # 黑金热血青春（静态，subtype=0）

```

```python
SRC_IC = 50125671  # 少年漫主角发型（动态，subtype=4）
```
