# 大模型接手指南 — FreeStyle 练习场发型恢复

> **目标**：让练习场（训练场）阶段也能显示 hook 替换后的发型。
> **当前状态**：sprintf 替换在房间阶段完全有效，但练习场走第二条路径绕过了 sprintf，恢复为原始发型。
> **生成日期**：2026-06-01

---

## 一、一句话任务

> 练习场阶段不调用 sprintf，直接通过 AcquireSMD 加载原始 ItemCode 的 SMD 资源。
> 需要找到练习场获取原始 ItemCode 的数据来源，并在该处拦截替换。

---

## 二、硬约束 — 绝对不能做的事

以下方案已全部实测失败（53 轮），**不可重试**：

### 🔴 游戏 .text 段 hook = 必崩

- `Interceptor.attach` 任何游戏 .text 地址 → Apollo 检测页属性变化(RX→RWX) → 杀进程
- 练习场资源加载密集，即使短时间不崩，进入练习场后必崩
- **唯一的例外**：冷函数 + rwx→hook→立即恢复 rx（见 `06_静态改动态不崩溃调试经验.md` §2.3），但只适用于极低频函数

### 🔴 Stalker = 必崩

- 全线程 Stalker → 游戏卡死
- 主线程 Stalker → 超时崩溃
- 不可用，不要尝试

### 🔴 ReadFile 替换 SMD 数据 = 必崩

- .pak 文件句柄被多线程共用 → 数据污染 → 崩溃或冻结
- 8 轮迭代全部失败

### 🔴 AcquireSMD hook 替换方向 = 死路（已确认）

- AcquireSMD 是高频函数，hook 即被 Apollo 检测
- SMD 资源路径在 AcquireSMD 调用前已被消费为 pack entry index，无法从外部替换
- **包含 13 次不同子方案的全部失败，不要在此方向浪费精力**

### 🔴 网络协议方向 = 死路

- TCP 加密（XOR + f12）已破解但改包无法影响外观
- UDP 走 AES 加密，密钥未破
- CryptEncrypt hook 零命中（游戏用内置 AES，不走 CryptoAPI）

### 🔴 内存直接修改 ItemCode = 无效

- 渲染管线使用缓存的模型数据，改 ItemCode 不影响已加载模型
- 暴力全量替换会破坏索引表

### 🔴 静态分析 .text 段 = 死路

- .text 段磁盘上加密（熵 7.23/8.0），Ghidra 反汇编输出为垃圾
- 非简单 XOR，流密码或块密码

### 🔴 DFileGPack::Open (0x1ACDF33) = 不可 hook

- Frida trampoline 对此函数不兼容，会崩溃

## 三、已确认的关键事实

### 3.1 练习场数据流（核心问题）

```
房间/过渡阶段:                          练习场实际渲染:
  sprintf("customize\item\i%d.xml",     ← hook 可替换 ✅    │  零 sprintf 调用 ❌
    50125461) → 替换为 50125711                              │
    → LoadItemFile → BML → AcquireSMD(50125711) ✅           │  AcquireSMD("res767\i50125461_FN.smd")
                                                            │  ← 原始 ItemCode！绕过 sprintf ❌
```

**结论**：练习场有第二条加载路径，不经过 sprintf，直接用原始 ItemCode 调 AcquireSMD。

### 3.2 内存状态

- SRC_IC (50125461) 在内存中**未被清零**：12/12 地址在练习场阶段仍含原始值
- `+0x7DC` 不是练习场渲染的数据来源
- 练习场阶段多了 1 个新地址含 SRC_IC

### 3.3 当前唯一成功方案（房间阶段）

| 步骤 | 方法 | 安全性 |
|------|------|--------|
| 1 | sprintf hook (MSVCR100.dll) 替换 ItemCode | ✅ 系统 DLL |
| 2 | `[ebp-0xD8]` 同步修改 | ✅ 栈操作 |
| 3 | 松散文件 `Resource\item\i50125711.bml`（XOR 0xFF 编码） | ✅ 无 hook |
| 4 | SeekFile 优先查松散文件 → 命中 | ✅ 无 hook |

### 3.4 Apollo 当前状态

- `sc stop ApolloProtect` 已执行
- ApolloCT.dll 已加载但 **0 保护线程**（CRC 已 patch）
- L2 ApolloShell 仍内嵌于 FreeStyle.exe，检测 .text 页属性变化
- **系统 DLL hook 安全**（sprintf/ReadFile/CreateFileA 等）

### 3.5 加载管线（源码级验证）

```
SetCharacterFeature(itemCode)
  → LoadItemFile → BML (XOR 0xFF → XML) → <mesh>/<texture> 路径
  → DGraphicAcquireSMD(path) → 读 SSKF magic → pMesh->Serialize()
  → BuildVertexBuffer() → 渲染
```

### 3.6 动态/静态发型机制

- CharacterMotion.bml 由**服务器下发**，每个 ItemCode 有 `Type` 字段
- Type → 工厂函数 → DDynamicActor 或 DStaticActor
- Actor 类型在创建时决定，不可后期转换
- 同类型替换（静态→静态 / 动态→动态）骨骼兼容；跨类型替换练习场必崩
- **但最新测试确认**：静态→静态替换在练习场也会恢复默认发型，说明问题不在跨类型

### 3.7 关键地址

| 名称 | RVA | 安全性 |
|------|-----|--------|
| DGraphicAcquireSMD | 0x01EEC130 | ❌ 游戏 .text |
| AcquireSMD 调用者 | 0x01EEBA30 | ❌ 游戏 .text |
| SetCharacterFeature(sprintf) | 0x01AE2563 | ❌ 游戏 .text（仅返回地址） |
| LoadItemFile | 0x01ACE1C0 | ❌ 游戏 .text |
| DFileGPack::Open | 0x1ACDF33 | ❌ 不可 hook |
| sprintf (MSVCR100) | 运行时查找 | ✅ 系统 DLL |
| ReadFile (kernel32) | 运行时查找 | ✅ 系统 DLL |
| CharacterMotion 解析 | 0x021B46E0 | ❌ 游戏 .text |

## 四、下一步推进方向（优先级排序）

来自 `07_练习场发型恢复根因.md` §5.3：

| 优先级 | 方向 | 方法 | 可行性分析 |
|:---:|------|------|-----------|
| **P0** | 找练习场专用加载函数 | hook 练习场场景切换，追踪角色加载调用链 | 需要找到练习场的触发点；系统 DLL hook 可用 |
| **P1** | 找装备数据缓存位置 | 内存扫描 ItemCode，看练习场阶段哪个地址被读取 | SRC_IC 12 个地址仍在，需要定位"谁在读" |
| **P2** | hook 角色创建函数 | 练习场创建角色时，追踪发型参数来源 | 游戏 .text hook 有风险，需冷函数+rwx恢复策略 |

### 可能的数据来源推测

1. **服务器下发的装备数据** — 进房间时服务器发送角色装备，练习场直接用这份缓存
2. **内存中的预加载缓存** — 房间阶段已加载的资源被缓存，练习场直接引用原始 ItemCode

### 可用工具/方法

- 系统 DLL hook：sprintf, ReadFile, CreateFileA/W（安全）
- 内存扫描：Frida Memory.readByteArray（只读安全）
- Pymem 外部读写（安全，不触 Apollo）
- Frida Interceptor + rwx→立即恢复 rx（仅冷函数）

## 七、BML 查找链（完整）

```
DFileGPack::Open(Group, Name)
  ├── 1. 松散文件: Resource\{Group}\{Name}     ← 优先！
  │    例: Resource\item\i50125711.bml
  ├── 2. 主 PAK: AcquirePack(Group) → {Group}.pak
  │    例: item768.pak
  └── 3. 扩展 PAK: AcquirePack(Group + "_ext") → {Group}_ext.pak
       例: item768_ext.pak
```

Group 计算规则：`item{编号}` — 具体编号需要 CreateFileA hook 确认或从 ItemCode 推断。
