### Part1 = Faces 数组 (SSkinTriangle)

```cpp
struct SSkinTriangle {
    byte  MaterialIndex;     // 1 byte
    word  WedgeIndex[3];     // 6 bytes
};  // = 7 bytes total
```

我们之前的 7-byte "edge record" `[uint16 a][uint16 b][uint16 c][uint8 sep]` 实际上是
`[byte MaterialIndex][uint16 WedgeIndex[0]][uint16 WedgeIndex[1]][uint16 WedgeIndex[2]]`，
因为起始偏移差了 1 字节导致字节对齐错位。

**WedgeIndex** 引用的是 Wedges 数组（FV section）的索引，不是 Points 索引。
所以"0%匹配 FV 三角形"是因为我们用错了索引空间。

### Part2 = Influences 数组 (SBoneInfluence)

```cpp
struct SBoneInfluence {
    float   Weight;          // 4 bytes
    int     BoneIndex;       // 4 bytes
    SVector LocalPos;        // 12 bytes
    SVector LocalNormal;     // 12 bytes
};  // = 32 bytes total
```

完全匹配我们之前发现的 32B/顶点记录。Weight=1.0, BoneIndex=8, LocalPos/LocalNormal = 骨骼局部空间坐标。

### Part3 = DProgressiveMesh 扩展字段

**不是预计算的三角形条带！** 是以下字段的总和：

```
[LodVertexNum][MinLodVertexNum][LodBias]      ← 12 bytes
[CtrlPoints array]                             ← 可变
[Sections array]                               ← 渲染批次描述
[InvisibleSections array]                      ← 可变
[CollisionMesh]                                ← 可变
[FaceLevels array]                             ← LOD 级别数据
[CollapseWedgeIndex array]                     ← 渐进网格折叠映射
[Blends array]                                 ← 骨骼蒙皮分组
[StaticBlend]                                  ← 静态蒙皮数据
```

**SMeshSection 结构** (序列化):
```cpp
int  MaterialIndex;       // 4B
int  BoneIndex;           // 4B
int  ObjectPriority;      // 4B
bool bShareIndexBuffer;   // 4B (dword序列化)
bool bVertexUpdate;       // 4B
bool bIndexUpdate;        // 4B
word MinIndex;            // 2B
word MaxIndex;            // 2B
word FirstFace;           // 2B
word TotalVerts;          // 2B
word TotalFaces;          // 2B
// = 34 bytes per section (不含VertexBuffer/IndexBuffer)
```

引擎使用 **D3DPT_TRIANGLELIST** (3索引/三角形)，不是 triangle strip。
BuildIndexBuffer 在运行时从 Faces 数组生成索引缓冲区。

## §65 加载管线完整分析 (源码确认)

### BML 加载 (Character.cpp)

```
SetCharacterFeature(itemCode)
  → sprintf("i%4d.xml", code)  → DCutPath/DCutExt → "i  50"
  → SFullName("item", "i  50.bml") → DFileGPack.Open()
  → LoadBinaryXML → 解析 <channel>, <character type="N"><object><mesh>,<texture>
  → RefreshItem() → DGraphicAcquireSMD(SFullName(path, filename))
```

**零校验** — 不验证 item code、mesh 路径、bone 数量。
Channel 系统是位掩码，重叠 channel 的物品会互相替换。

Mesh type (BML `<type>` 字段):
- `normal` = 0: 普通网格
- `fskin` = 2: 面部皮肤 (合成纹理)
- `uskin` = 3: 上身皮肤 (合成纹理)
- `lskin` = 4: 下身皮肤 (合成纹理)

### SMD 加载 (DGraphicObj.cpp)

```
DGraphicAcquireSMD()
  → 读4字节 → "SSKF"?
    YES: 读整个文件到内存 → pMesh->Serialize(reader)
    NO:  当旧版SMD解析 → pMesh->ImportOtherMesh()
  → pMesh->BuildVertexBuffer()  ← GPU缓冲区运行时生成
  → 加载纹理
```

**零校验** — 不检查 bone count、vertex count、mesh_flag。
Serialize 纯顺序反序列化。

### GPack (.pak) 格式

```
文件末尾: SUC block { "SUCI", version, PackHeaderPos, PrevSUCPos }
PackHeader: "NFGP" + 元素目录
  → 元素: {name, offset, size} 按名称排序，二分查找
```
**无校验和、无加密、无压缩。**

### 替换约束

1. **Bone index 必须兼容** — UpdateMesh 用 `aBoneMap[InfluencePtr->BoneIndex]` 映射骨骼。
   如果源/目标骨骼表不同，蒙皮会变形。
   同角色类型（男/女）的骨骼表一致。

2. **文件大小不影响** — GPack 按 name+offset+size 定位，不要求固定大小。
   但替换后需要重建 pak 头部（更新 offset 和 size）。

3. **Part3 (PM扩展) 不是问题** — 即使渐进网格数据对新模型不匹配，
   最坏情况是 LOD 级别不正确，不会导致崩溃或光头。

### 最安全的替换方案

**dynamic→dynamic, 同角色类型**: 直接替换 SMD 文件内容。
骨骼表兼容（Bip01 hierarchy），Influences 的 BoneIndex 直接有效。

### 风险 1：骨骼映射是按名称匹配 (关键)

**源码** (`DSkeleton.cpp:566`):
```cpp
void DActor::BuildBoneIndex( int* Indices, SName* Names, int Num )
{
    For( int i = 0; i < Num; ++i )
    {
        Indices[i] = -1;
        For( int j = 0; j < m_Skeleton.Bones.Num(); ++j )
        {
            if( Names[i] == m_Skeleton.Bones[j].Name )
            {
                Indices[i] = j;
                break;
            }
        }
    }
}
```

**映射流程**:
1. `SetCharacterType(type, pAnimation)` → `AssignAnim(pAnimation)` → 动画文件的骨骼表成为角色主骨骼
2. `AttachSMD(handle)` → `RebuildRenderTemplate()` → 用 SMD 内嵌骨骼名与主骨骼名逐一匹配
3. `UpdateMesh()` → `aBoneMap[SMD_bone_idx]` 查到 actor_bone_idx → 用 `SkelMatrices[actor_bone_idx]` 变形顶点
4. 匹配失败 → `aBoneMap[i] = -1` → `UpdateMesh` return false → **角色不可见**

**每个 SMD 自带 DSkeleton (含骨骼名列表)**。Influences 中的 BoneIndex 是指向 SMD 自身骨骼表的索引。
引擎先读 SMD 骨骼名 → 按 name 在角色主骨骼中查找 → 得到映射表。

**安全性**: 所有游戏内 SMD 文件共用 Bip01 骨骼层次结构，名称天然匹配。
**同角色类型 (男→男 / 女→女) 100% 安全**。
**跨角色类型**: 男女共用 Bip01 骨骼名，理论安全，但未实测。

**PAK 格式细节** (DFileGPack.h):
```
文件头 (NFGP):
  dword dwIdentifier = 0x4E464750 ("NFGP")
  dword dwVer = 2
  dword dwCryptMethod (保留=0)
  dword dwRes1, dwRes2
  dword dwNumShares + 共享名列表

元素头 (每个文件):
  dword dwNameSize          (含 null)
  dword dwElementPos        (数据绝对偏移)
  dword dwElementSize       (数据大小)
  dword dwNextElement       (下一元素头偏移, 0=链尾)
  char  szName[dwNameSize]  (小写, null-terminated)
  [数据在 dwElementPos 处]

文件尾 (SUC):
  dword = 0x53554349 ("SUCI")

  dword PrevSUCPos (同值=第一层)
```

SUC 链支持增量补丁：新 pak 头追加到文件末尾，新 SUC 指向新头。

## §70 二进制 SSKF v1 格式验证 (实际文件分析)

**关键发现：源码文档 (`SSKF文件格式文档.md`) 与实际二进制存在多处不匹配。**

### 70.1 版本号：源码说 v0，实际是 v1

所有 4 个 SMD 文件 (i50125461_MT, i50125671_MT, i50125331_MT, i50125711_MT) 的 version 字段均为 `01 00 00 00` (=1)。

源码 `SKIN_LASTEST_VERSION = 0` 显然已过时，实际游戏使用的版本更高。

影响：**v1 的 SBone 包含 Scale 字段**（最后 12 字节），固定字段部分共 56 字节。

### 70.2 SName 存储格式：不是 FString，是 null-term ASCII

源码文档说 SName 是 `[int length][char data]`（FString 格式），但实际二进制：

- Mesh Name：直接 null-terminated ASCII，4 字节对齐
  ```
  偏移 0x08: 69 31 32... 00 00 00 ("i50125671_FN\0" + padding)
  ```
- Skeleton Name (Group)：同样是 null-term ASCII + 4 字节对齐
  ```
  None\0 + padding
  ```
- Bone Name：同上

**这不是 FName (Index+Number=8B)，也不是 FString (length prefix)**，是纯粹的 null-terminated ASCII + 4B align。

### 70.3 骨骼表标记：0x75 + int32 count

在第一个骨骼名 "Bip01" 前 5 字节存在固定标记模式：
```
75 XX XX XX XX     ← 0x75 + int32 count
```
count = 实际骨骼数 + 1（包含骨架自身 "None" 条目）。

例如 i50125461_MT (75 骨骼)：count = 76 (0x4C)
例如 i50125711_MT (78 骨骼)：count = 79 (0x4F)

每个 SBone 起始于 4 字节对齐边界，结构为：
```
[SName Name] + [SName Group] + [ParentIndex 4B] + [NumChildren 4B] + [Depth 4B] + [Flags 4B] + [Pos 12B] + [Quat 16B] + [Scale 12B, v≤1]
= 可变(Name+Group) + 56B 固定字段
```

### 70.4 4 个 SMD 骨骼数确认

| 物品 | 类型 | 实际骨骼数 | 表头 count |
|------|------|-----------|-----------|
| i50125461_MT | 静态借用 | 75 | 76 (0x4C) |
| i50125671_MT | 动态 | 73 | 74 (0x4A) |

### 71.3 真实的动态/静态区别在 Actor 对象

根据 [调试总结.md §9](file:///D:/py/反编译/FreeStyle/docs/07_动态发型功能/调试总结.md) 已有分析：

**动态/静态区别不在描述符中**，而在运行时创建的 **Actor/MotionObject** 中：

| 项目 | DDynamicActor | DStaticActor |
|------|--------------|-------------|
| vtable | 0x284AA4C | 0x284E114 |
| 运动类型 (+0x0C0) | 11 (Dynamic) | 12 (Static) |
| 物理标志 (+0x74) | &0x18 非零 | &0x18 = 0 |
| 物理参数 (+0x460) | 非零 | 零 |
| 初始化函数 | DynamicInit (0x229B4D0) | StaticInit (0x24C5910) |

**决策链**：
```
CharacterMotion.bml (服务器下发)
  → 解析函数 0x21B46E0 读取 ItemCode%d 条目
  → Type 字段值决定创建哪个 Actor 类型
  → 工厂函数 0x021C1F00 根据 vtable 指针创建对象
  → DDynamicActor 有物理初始化 ∨ DStaticActor 无物理初始化
```

### 71.4 分析影响

1. **+0x058 在描述符中 = 物品子类型**，与物理效果无关
2. **动态/静态控制不在 SMD 或 BML 文件中**，在 CharacterMotion.bml（服务器下发）
3. **运行时修改**是唯一途径：Hook 解析函数或构造函数，替换 Actor 类型

| subtype | 物品 | 类型 |
|---------|------|------|
| 0 | 黑金热血青春发型 | 静态 |
| 1 | 紫色护目镜马尾辫 | 静态 |
| 2 | 紫色超赛发型 | 动态 |
| 3 | 粉色超级赛亚人3 | 静态 |
| 4 | 少年漫主角发型 | 动态 |
| 5 | 少年漫劲敌发型 | 动态 |

→ **subtype 是发型变体编号 (0~5)，与动态/静态无关。** 之前的 "2 vs 0" 对比只是两个具体条目的 subtype 差异。

### 76.2 真实的动态/静态区别

根据 [调试总结.md §9](file:///D:/py/反编译/FreeStyle/docs/07_动态发型功能/调试总结.md) 已有记录：
- 动态/静态在 **Actor 对象层**（DDynamicActor vs DStaticActor）
- 由 CharacterMotion.bml（服务器下发）驱动
- Actor vtable: DDynamicActor=0x0284AA4C, DStaticActor=0x0284E114
- Actor+0x0C0: motion type (11=Dynamic, 12=Static)
- Actor+0x074: flags (&0x18 启用物理)

### 76.3 多次内存扫描结论

| 尝试 | 脚本 | 结果 | 结论 |
|------|------|------|------|
| Actor vtable 全量扫描 | scan_actor_vtable.py | DDynamic:1(假), DStatic:0 | Actor 对象不存在于大厅 |
| 描述符 vtable 全量扫描 | quick_vtable_test.py | VT_A0:1433(全假) | vtable 模式在代码段大量出现 |

**核心结论：**
1. **Actor 对象 (DDynamicActor/DStaticActor) 在大厅状态下不存在**，只有进游戏/角色预览才创建
2. **描述符对象**在 Frida 扫描中存在 949 个，但通过 Pymem 扫描无法定位（可能在 PAGE_EXECUTE_READWRITE 或其他保护类型的内存中）
3. **vtable 模式扫描法无效** — vtable 地址值在代码段、rdata 段大量出现，无法区分对象实例

### 76.4 发现的 Item 映射表

位于 `0x222CC8F0`，每条 0x30 字节，结构：
```
+0x00: ItemCode (int32)
+0x04: ptr1 → 物品配置结构 (含GBK名称、子分类等)
+0x08: ptr2 = ptr1 + 0x8C (同一结构的子字段)
+0x0C: ptr3 = ptr2
+0x10: 0x00000000 (padding)
+0x14: ItemCode 字符串 (12 bytes, ASCII)
+0x20: padding
+0x24: 0x00000008 (固定)
+0x28: 0x0000000F (固定)
```

ptr1 指向的物品配置结构：
```
+0x00: ItemCode 字符串 (8 bytes + null)
+0x08: padding
+0x0C: 08 00 00 00 (固定)
+0x10: 0f 00 00 00 (固定)
+0x18: "768" 或类似数值字符串
+0x28: 子分类 (3=发型, 4=?, 0F=?)
+0x34: 索引号 (2147~2149)
+0x40: 子分类2 (3/4/0F/10)
+0x48: GBK-encoded 物品名称
```
