## 二、5.19 新增确认的技术事实

### 2.1 ✅ E2 入口不接收 game packet buffer

**Step 0 诊断结果：37 个包全部 ORPHAN**

- `step0_diag.py` / `frida_step0_diag.js`：同时 hook E2 和 E0 onEnter，对比 buffer 地址
- 结果：E0 onEnter 拿到 37 个 game buffer，E2 onEnter **零匹配**
- 再对 E2 入口做参数诊断（`frida_e2_callsite.js` 的 E2_ENTRY 日志）：

```
[E2_ENTRY#1] ecx=0x28B68C10 edx=0x00333631 ret=0x0243886E tl4=0
[E2_ENTRY#2] ecx=0x28B68C10 edx=0x28B70CD4 ret=0x02439276 tl4=0
[E2_ENTRY#3] ecx=0x28B68C10 edx=0x000063AA ret=0x0242FB99 tl4=0
...
```

- `tl4=0` — E2 收到零长度 WSABUF，不包含游戏包数据
- `ecx` 固定为 session 对象指针，`edx` 每次不同，`ret` 来自多个调用者
- **结论：E2 内部自行构造 buffer → 写入 field12 → 传给 E0，不是从参数拿现成 buffer**

### 2.2 ✅ E0 call site RVA 确认 = 0x2038E14, E2 内偏移 = 0x294

从 `dynamic_trace.txt` 的回溯：
```
#2 0x2438e19 [FreeStyle.exe+0x2038e19]  ← E0 的返回地址
```

CALL 指令在返回地址前 5 字节：`0x2038E19 - 5 = 0x2038E14`

偏移：`0x2038E14 - 0x2038B80 = 0x294`

**之前两次算错**（0x699→0x694），导致 hook 了野地址（opcode=0xF0 LOCK 前缀），游戏崩溃。现已修正并加了 opcode 验证（非 0xE8 拒绝 hook）。

### 2.3 ✅ 明确了 field12 写入窗口

```
field12 写入窗口：
  [E2 onEnter]  ← E2 开始执行，buffer 尚未构造，field12 尚未写入
       │
       ▼  ← ← ← field12 在此区间被写入 ← ← ←
       │
  [E2+0x294]    ← CALL E0，buffer 已构造完毕，field12 已写入
       │
       ▼
  [E0 onEnter]  ← WSASend 包装器，不做任何处理
```

**确认：field12 在 E2 函数体内的 0x294 字节指令区间内生成。**

## 三、5.19 尝试过的方案及结果

### 3.1 ❌ e2_checkpoints.py — CALL 指令级 Interceptor 二分法

**方案：** 反汇编 E2 内所有 CALL，hook 每个检查 field12 变更点。

**第一轮 — hook ALL CALLs + AFTER (每条CALL的下一条指令)：**
- 912 条指令，60 个 CALL，hook 了 45 个 E8 + 45 个 AFTER = 90 个 Interceptor
- 结果：**ACCESS_VIOLATION 崩溃** (exit code 0xC0000005)
- 原因：FF xx 短指令被 Frida 5 字节 trampoline 覆盖邻接代码；AFTER 指令可能是循环跳转目标

**第二轮 — 只 hook E8 near call，跳过 AFTER + FF：**
- 912 条指令，只 hook 45 个 5 字节 E8 CALL
- 结果：游戏启动但 **"未响应"**，E2 被调用了但 E2 onEnter 未匹配 magic
- 原因：E2 入口不接收 game buffer（magic 检测失败），且 45 个 hook 对性能影响大

### 3.2 ⚠️ e2_callsite.py — hook E2 内部 CALL E0 指令

**方案：** 只 hook 一个点 — E2+0x294（CALL E0），读 `[esp]` 拿 WSABUF → buffer → field12。

**第一轮 (offset=0x699, 算错)：**
- opcode=0xF0 (LOCK prefix)，非 0xE8 → opcode 验证拒绝 hook → **没崩但没数据**

**第二轮 (offset=0x694, 算错)：**
- `0x2038E19 - 5 = 0x2038E14`，但错误地认为 E2 偏移 = 0x2038E19 - 0x2038B80 = 0x699
- 实际偏移 = 0x2038E19 - 0x2038B80 = 0x299, CALL = 0x299-5 = 0x294
- 差了 0x400 → hook 了未知地址 → **游戏启动崩溃**

**第三轮 (offset=0x294, 正确)：**
- @0x2038E14 的 opcode 应为 0xE8
- 即使正确，hook E2 内部指令在启动阶段 E2 被调用时崩溃
- 原因：E2 在游戏初始化阶段被多次调用（已知 10 次），trampoline 在函数体内位置敏感

### 3.3 ❌ e2_stalker.py — Frida Stalker 指令级追踪

**方案：** 用 Stalker（不改代码，复制 basic block 到新内存 + 插 callout）替代 Interceptor：
1. E2 onEnter → 启动 Stalker.follow(threadId)
2. Stalker transform: 在 E2+0x294 (CALL E0) 插 callout 读 WSABUF→field12
3. Stalker transform: 在每个 MOV-to-memory 指令插 callout 记录 CPU 状态
4. E2 onLeave → 停止 Stalker, dump 所有 store 记录

**结果：** **零输出**（仅 `Game exited=0`）

- 文件：`frida_e2_stalker.js` + `e2_stalker.py`
- 连 `JS loaded` 都没有 → JS 加载阶段失败
- 可能原因：
  - `Stalker.follow()` API 在该 Frida 版本中不可用或签名不同
  - JS 运行时语法错误未被 send() 捕获（stderr 不可见）
  - Frida + Windows x86 + Stalker 兼容性问题

### 3.4 ✅ offline_f12.py — 离线算法分析

**方案：** 对 `correct_buf.txt` 的 38 个包做算法穷举。

**结果：全部标准算法 0 匹配**
```
CRC32 / Adler32 / Fletcher32 / XOR / SUM32 / Chained CRC32 → 全挂
f12 XOR enc[0:4] → 38 个唯一值
f12 XOR enc[4:8] → 38 个唯一值
```

**结论：field12 是自定义算法。**

## 五、当前核心阻塞与务实评估

### 5.1 唯一卡点：field12 算法

已知：
- field12 在 E2 函数体内生成（E2 入口→E2+0x294 之间）
- 所有标准算法（CRC/Adler/Fletcher/XOR/SUM）全挂

未知：

### 5.2 能工作的手段 vs 不能工作的手段

| | 能工作 | 不能工作 |
|---|---|---|
| **Frida** | hook 函数入口(WSASend/E0/E2) | hook 函数体内指令 |

### 5.3 卡住的本质

我们需要看见 E2 函数体内部发生了什么，但 Frida 无法安全地在 E2 体内插入探针。能 hook 函数入口，但入口拿不到我们关心的中间状态。

## 十一、🚀 重大突破：field12 plen=8 算法已完全破解 (5.19 深夜)

### 11.1 背景

通过 Frida 热补丁方式绕过了 Apollo 完整性检查：
1. 恢复原始 `ApolloCT.dll` 到磁盘 → T2BOX 正常启动
2. Frida attach 后用 `Memory.protect` + `writeByteArray` 在内存中 hotpatch ApolloCT CRC
3. 然后 Interceptor.attach WSASend 抓包

捕获了三组 session 数据：
- **S1 (OLD)**: 38 个包 (`correct_buf.txt`)，seq 1-38
- **S2 (NEW)**: 22 个包，seq 16-37（后来被覆盖）
- **S3 (NEW)**: **102 个包**，其中 **79 个 plen=8**，seq 48-149

### 11.2 关键发现：跨 session 的 XOR 一致性

S1 和 S2 有 7 个重叠 seq (16,24,27,28,31,32,33)，**所有重叠 seq 的 f12 XOR = `0x9EDEA41A`**，尽管 b0 差异从 -8 到 +40 完全不同。

→ 这意味着：`f12(b0, s) = SessionKey XOR F(b0, s)`，F 与 session 无关。

### 11.3 核心突破：U[seq] 交替模式

S3 中 `b0=0x00, seq=74→75` 的 dF = `0xDD45AAB8`，且 b0 相同、ds=1。

检验所有相邻 seq (ds=1): **f12 XOR 恒为 `0xDD45AAB8`**。

→ `U[s+1] = U[s] XOR 0xDD45AAB8`（对所有 s 恒定）

### 11.4 核心突破：T[b0] 的 4-byte 组结构

S3 的 seq 100-147 是 48 个连续的数据，b0 按 nibble 成群：

```
seq 108→109: b0 0xA3→0xA2, dF = 0xDD45AAB8 (odd→even, T pair 内)
seq 109→110: b0 0xA2→0xA1, dF = 0x62228939 (even→odd, T cross-pair)
seq 110→111: b0 0xA1→0xA0, dF = 0xDD45AAB8 (odd→even, T pair 内)
```

跨越 4-group (同低 2 bit):
```
T[0xA3] XOR T[0xA7] = 0x6402E328  (通过 seq 108 vs 112, 均 even)
T[0xA2] XOR T[0xA6] = 0x6402E328  (通过 seq 109 vs 113)
```

**所有 nibble 组 (0xAx, 0xBx, 0x8x, 0xDx) 都遵循同一模式！**

### 11.5 完整算法

```
f12(b0, seq) =  SessionKey
             XOR  G[b0 >> 2]                        // 64-entry nibble-group表
             XOR  ((b0 & 2) ? 0xBF672381 : 0)       // T pair differentiation
             XOR  ((seq & 1) ? 0xDD45AAB8 : 0)      // U seq parity
```

其中：
| 符号 | 含义 | 值 |
|------|------|----|
| `SessionKey` | 每 session 随机的 32-bit 常量 | 通过抓1个包反推 |
| `G[n]` (n=0..63) | 64-entry 表，n = b0>>2 | 可用 S3 数据填充 |
| `0xBF672381` | T 的 pair 间差分常量 | 固定 |
| `0xDD45AAB8` | U 的 seq 交替常量 | 固定 |

### 11.6 验证结果

- **结构化阶段 (seq 74-147)**: `67/67 = 100%` ✅
- **初始化阶段 (seq 50-71)**: `0/12` ❌ (可能使用不同的 U 或 SK)

12 个失败全部在 seq 50-71 的登录/选角阶段，**游戏大厅及之后的所有包 100% 匹配**。

### 11.7 实用价值

在同一 session 内构造伪造 plen=8 包的方法：

```python
# 1. 抓1个已知包 → 反推 SessionKey
session_key = f12_known ^ G[b0_known >> 2]
             ^ (0xBF672381 if b0_known & 2 else 0)
             ^ (0xDD45AAB8 if seq_known & 1 else 0)

# 2. 构造新包的 field12
f12_new = session_key ^ G[b0_new >> 2]
          ^ (0xBF672381 if b0_new & 2 else 0)
          ^ (0xDD45AAB8 if seq_new & 1 else 0)
```

## 十二、S4 数据分析：T[b0] 表结构深化 (5.19 深夜续)

### 12.1 S4 数据概况

- 133 条记录，88 个 plen=8
- 游戏崩溃于 seq=150-151（练习场走动时 `data failed`）
- 可能原因：Apollo 时序检测 / 网络 seq 错位 / 新 b0 值未在表中

### 12.2 T[b0] 内部结构的完整发现

S4 每个 nibble-2 组内有 **3 个不同 G 值**（而 S3 每组只有 1 个），这意味着之前的模型 `G[n2]` 不完整 — 实际是完整的 **T[b0] 256 条目表**（每个 b0 值对应一个 32-bit 值）。

合并 S3+S4 → **114/256 条目已填充**。

### 12.3 T[b0] 根据 b0 bit7 分为两种模式

**模式 A** (b0 < 0x80，低半区)：
```
T[4k]   == T[4k+3]                            (位置 0 和 3 相同)
T[4k+2] == T[4k] ^ 0xBF672381                  (位置 2 = K_X 偏移)
T[4k+1] == T[4k] ^ D[k]                        (位置 1 = 组相关偏移)
```

**模式 B** (b0 >= 0x80，高半区)：
```
T[4k]   == T[4k+1]                             (相邻对 0=1)
T[4k+2] == T[4k+3] == T[4k] ^ 0xBF672381       (相邻对 2=3, 且 = 基值^K_X)
D = 0                                           (无额外偏移)
```

### 12.4 D[k] 值的分布规律

仅在模式 A 中存在，遵循 `k mod 4`：
- k mod 4 = 1 或 3：D = `0xC4451272`
- k mod 4 = 2：D = `0x32017194`
- k mod 4 = 0：D 随 k 变化（k=4: `0x0C40D422`, k=8: `0xDB65C0A9`, k≥0x20: `0`）

### 12.5 plen=8 包的内容一致性

同 b0 的包 dec 内容 100% 一致（114/115），b0 自身就是 packet type：
- `b0=0x00` → 心跳包 `4db8a85447b8a854`
- `b0=0x01` → 另一固定短包 `4c90a85b10b8d5a0`

**结论：plen=8 包已可完整构造和伪造。**

## 二十一、T[b0] 表补全尝试与重大修正 (22:40-23:10)

### 21.1 合并 S1-S5 全部 plen=8 包

| Session | 总包 | plen=8 | unique b0 |
|---------|------|--------|-----------|
| S1 | 38 | 18 | 17 |
| S3 | 102 | 79 | 71 |
| S4 | 133 | 88 | 86 |
| S5 | 146 | 93 | 82 |
| **合计** | **521** | **357** | — |

合并后经验填充 **163/256 (63.7%)**，模式推断后 **192/256 (75.0%)**。

### 21.2 关键发现：field12 依赖 payload 内容

跨 session 对齐测试揭示重大问题：

```
S4 ↔ S5: 57 共享 b0，仅 17 个可通过固定偏移匹配 (30%)
S3 ↔ S5: 17 共享 b0，仅 1 个匹配 (6%)
```

更严重的是 **同一 session 内同一 b0 有多个不同的 sk_xor_t 值**：

```
S5 b0=0x00: 5 种不同值
S4 b0=0x00: 3 种不同值
S3 b0=0xD5: 2 种不同值
```

**原因**：seq 1 和 seq 2 同为 b0=0x00 心跳包，但 decrypted payload 不同（`4db8a85447b8a854` vs `4db8a85441b8a854`），field12 也不同。

### 21.3 修正后的 field12 模型

```
field12(payload_bytes, seq) = F(payload) XOR ((seq & 1) ? U_PARITY : 0)
```

- `F()` 是整个 payload 内容的校验/签名函数，不仅是 b0
- `U_PARITY = 0xDD45AAB8` 仍是 seq 奇偶常量
- b0 只是 packet type 分类标签，不决定 field12
- 之前 67/67 验证通过是因为那 67 个包的 payload 恰好遵循固定模式（游戏大厅阶段动作包）

### 21.4 T[b0] 表的实际定位

T[b0] 表作为"同 b0 固定 payload 对应的 field12 基础值"仍有参考价值：
- 163 个 b0 有经验值
- 缺失 64 个 b0 集中在 16 个完整 nibble group（对应未触发的游戏操作）

### 21.5 结论

**仅靠出站数据无法完全破解 field12。**
需要入站数据做以下验证：
1. 入站包的字段结构和出站是否完全一致
2. 服务端回复包是否使用相同的 F(payload) 函数
3. 多 session 数据做差分分析，分离 session-dependent 和 payload-dependent 成分
