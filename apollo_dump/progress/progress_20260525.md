## §54 Post-FV Section 完整结构解码 (SSKF/SMD 格式核心突破)

### 三段式布局

```
Post-FV = Part1 (edge/adjacency) + Part2 (per-vertex 32B) + Part3 (submesh header + tri indices)
```

| 部分 | Static (489v, 196t) | Dynamic (1547v, 554t) | 内容 |
|------|--------------------|-----------------------|------|
| **Part 1** | 5454B (11.2B/v, 27.8B/t) | 18247B (11.8B/v, 32.9B/t) | 边/邻接三角形数据 |
| **Part 2** | 15648B = 489×32 | 49504B = 1547×32 | Per-vertex 32 字节记录 |
| **Part 3** | 2812B (5.8B/v, 14.3B/t) | 8620B (5.6B/v, 15.6B/t) | 子网格头 + 三角形索引 |
| **合计** | **23914B** | **76371B** | |

### Part 1: 边/邻接三角形数据

- **起始**: 2×float (UV 坐标?) + uint16 (vert_count-1) + 变长前缀
- **主体**: 7-byte stride 记录
  - Static offset=1: 71% valid, offset=6: 77% valid
  - Dynamic offset=3: 95.5% valid, offset=5: 94.7% valid
- **验证**: §52 的 "成对 pattern" 在此 — 共享边的三角形对

### Part 2: Per-vertex 32 字节记录

**每条记录格式 (32 bytes, 8 个 float):**

```
[+00] float: blend/weight 值
       - v0 = vert_count (0x01E9=489 for static, 0x060B=1547 for dynamic)
       - 其余顶点: 范围 [-0.95, 1.06]
[+04] float: 1.0 (常量标记)
[+08] float/uint32: 0.0 (raw=0x00000008, 可能是 uint16 count=8 + uint16=0)
[+0C] float[0]: avg=0.15, mostly positive (484/489), range [-0.02, 0.26]
[+10] float[1]: avg=0.00, balanced pos/neg (243/246), range [-0.11, 0.11]
[+14] float[2]: avg=-0.05, more negative (140/349), range [-0.15, 0.11]
[+18] float[3]: avg=0.24, mostly positive (320/489), range [-0.83, 1.24]
[+1C] float[4]: avg=-0.08, more negative (214/275), range [-1.10, 1.09]
```

**特征:**
- 恰好 vert_count 条记录 (static=489, dynamic=1547)
- 间距精确 32 字节 (100% 一致性)
- 1.0f 标记精确出现 vert_count+1 次
- count 字段始终 = 8
- float[3..7] 包含正值和负值, 疑似 tangent/blend space 或 skinning 权重
- float[3] + float[6] 的分布最宽 (up to ±1.2), 可能是主 tangent 分量

### Part 3: 子网格头 + 三角形索引

**头部结构 (静态前 44 字节):**
```
[float: 0.93]          — 参数 (bounds max?)
[uint32: 489]           — vert_count
[uint32: 48]            — submesh/bone group count
[float: 1.0]            — 标记
[uint32: 0] [uint32: 1] [uint32: 0] [uint32: -1]
[uint32: 0] [uint16: 256] [uint16: 1] ...
```

**三角形索引列表:**
- uint16 × 3 格式
- **Static: 97.3% valid**, Dynamic: **98.2% valid**
- 每索引 < vert_count

## §55 完整 SSKF (SMD) 文件格式 v2 (更新 §51)

```
0x0000 - 0x0147  Header (328 bytes)
  +0x00: "SSKF" magic (4B)
  +0x04: version=1 (4B)
  +0x08: mesh name (64B)
  +0x48: parent bone name (64B)
  +0x88: flag byte (0x00)
  +0x89: dataOff (uint32, varies: 0x3B75~0x4E75)
  +0x8E: "Bip01" marker (64B)
  +0xCE: ".None" parent reference
  +0x10E: uint8 0x75
  +0x110: uint32 0xFFFFFFFF
  +0x114: uint32 0x02
  +0x120: 3×float (bounding box? ~[0.86, 0.07, 0.18])
  +0x12C: 3×float ([~0.18, 0.18, 1.0])
  +0x138: uint32 0x01
  +0x13C: uint32 0x00894350
  +0x140: uint16 bone_count_minus_1
  +0x142: uint16 ???
  +0x144: "Bip01 " (start of first bone name)

0x0148 - Bone table (variable count × 0xBA stride)
  Per bone (186 bytes):
    +0x00: name (64B, null-terminated)
    +0x40: transform data (122B)
      +0x00: parent name fragment ("nNone", ".None" etc)
      +0x40: uint8 0x75
      +0x42: uint16 flags
      +0x44: uint32 bone_index
      +0x48: float: rotation component (-980.0 for root = flag?)
      +0x4C-0x58: 3×float rotation quaternion
      +0x58: float 1.0
      +0x5C: uint32 0x00894350
      +0x60: uint16 next_bone_index
      +0x62: uint16 ???
      +0x64: next bone name ("Bip01 ...")

  Bip01 bones: 57 (标准 3ds Max Biped)
  Helper bones: variable (Finger2Nub, Finger3, etc.)

~bone_end - Material section
  "Material #NNNNN" (null-terminated)
  texture path (null-terminated)

~0x2BFC: flag byte (0x46 static / 0x6E dynamic)
~0x2BFD: uint16 vertex count
~0x2BFF: uint16 marker (0 or 1)

0x2C01 - Position channel (vc × 12B = vec3 float, SoA)
Position_end - Normal channel (vc × 12B = vec3 float, SoA)

Normal_end - FV UV+Idx block (stride=10B)
  Per face-vertex: [float U(4B)] [float V(4B)] [uint16 vertex_index(2B)]
  Static: 588 entries = 196 triangles
  Dynamic: 1664 entries = 554 triangles + 2 remainder

FV_end - Post-FV section:
  Part1: edge records (~11B/vert, 7-byte stride per face-vertex)
  Part2: per-vertex records (vc × 32B, 1.0f+count=8+5×float)
  Part3: 76B header + triangle strip uint16 index data

EOF-8~12: Index buffer header [uint32=1][uint16=vc]
  + Index buffer (vc × uint16, 0..vc-1)
  + Trailing zeros
```

### dataOff 含义 (部分解决)

| 文件 | dataOff | bone_count | 差值 |
|------|---------|-----------|------|
| static (res767) | 0x3B75 | 57+18=75 | ? |
| dynamic (res767) | 0x3B75 | 57+18=75 | 同 |
| smd_compare _MT | 0x4C75 | ~122 | 更大 |
| smd_compare _FN | 0x4E75 | ~120 | 更大 |

dataOff 不直接等于任何 section 起始偏移。可能与文件总数据区偏移有关。

### flag byte 0x46 vs 0x6E

- 0x46 = ASCII 'F' → Fixed/Static
- 0x6E = ASCII 'n' → ? (dynamic?)
- smd_compare 文件的 flag 全为 0x00 → 可能是提取/转换时丢失

## §57 Part3 Triangle Strip 解码 (新增 0525-B)

### Part3 完整结构

```
Part3 = 76-byte header + triangle strip index data
```

### 76-byte Header

```
+0x00: float bounds_param (static=0.935, dynamic=1.065)
+0x04: uint32 vert_count (489 / 1547)
+0x08: uint32 submesh_count (48 / 154) ← 不是常数！
+0x0C: float 1.0
+0x10: uint32 0
+0x14: uint32 1
+0x18: uint32 0
+0x1C: uint32 -1 (0xFFFFFFFF)
+0x20: uint32 0
+0x24: uint32 0x00010100 (uint16=256, uint16=1)
+0x28: uint32 varies (含 FV count 相关值)
+0x2C: uint32 varies
+0x30: uint32 3(static)/10(dynamic) (material group?)
+0x34: uint32 256
+0x38-0x47: zeros
+0x48: uint32 varies (含 strip data length)
+0x4C: uint16[] triangle strip data starts
```

### Triangle Strip 数据

**格式**: 连续 uint16 vertex indices, 构成 triangle strip
**Restart**: 退化三角形 (连续两个相同 index) 分隔不同 strip
**Strip 数量**: static=49, dynamic=90 (submesh_count: 48, 154)

| 指标 | Static | Dynamic |
|------|--------|---------|
| Strip count | 49 (expected 48) | 90 (expected 154) |
| Total strip triangles | 1169 | 3906 |
| Index range | 0-489 (5 OOB) | 0-1664 (4 OOB) |
| Validity | 99.6% | 99.9% |
| Strip sizes | 3-94v (avg 26) | 3-455v (avg 45) |

**关键特征**:
- Strip 范围高度重叠 (1073/1176 pairs), 不是简单的 bone group 范围划分
- Part2 float[3] (Y 相关) 在不同 strip 之间有显著变化 (0.14~0.74)
- Part3 三角形与 FV 三角形几乎不重叠 (0-0.5%), 是完全独立的几何数据
- 无 FV vertex index mapping — Part3 纯粹是 triangle strip 渲染索引

### submesh_count 含义

- 48 (static) ≠ 57 (Bip01 bones), 但接近
- 154 (dynamic) >> 57, 可能与 skinning 影响的骨骼数量相关
- 与实际 strip 数量不完全匹配, 可能有空 submesh 或 strip 合并

## §59 Part2 Per-Vertex Record 完整解码 (更新 0525-D)

### 核心突破: Part2 = 骨骼局部空间的位置+法线

2D 多变量回归得到 **R²=1.0** (static/dynamic 系数完全一致):

```
f0 = cos(12°)*Y - sin(12°)*Z - 1.546    R²=1.000000
f2 = -sin(12°)*Y - cos(12°)*Z + 0.359   R²=1.000000
```

其中 cos(12°)=0.978148, sin(12°)=0.207910 — **精确的12°绕X轴旋转**

### 坐标变换解码

```
世界空间 → 骨骼局部空间 (12°绕X旋转):
  X_bone = f1 = X (不变)
  Y_bone = f0 + 1.546 = cos(12°)*Y - sin(12°)*Z
  Z_bone = -(f2 - 0.359) = sin(12°)*Y + cos(12°)*Z

骨骼世界位置 ≈ (0, 1.587, 0.030)  (头部/颈部高度)
```

### 法线编码

```
pre = -1.086*NX + 0.011*NY - 0.058*NZ - 0.047   R²=0.97  (≈ -NX)
f3  = -0.040*NX + 0.007*NY + 1.099*NZ + 0.095    R²=0.92  (≈ NZ)
f4  = -0.002*NX + 1.089*NY - 0.003*NZ + 0.006     R²=0.998 (≈ NY)
```

法线未经过12°旋转 (位置精确旋转, 法线近似存储)

### 完整记录格式 (32 bytes)

```
[+00] float: 骨骼局部 NX*1.09 (v0 = vertex_count 特殊标记)
[+04] float: 1.0 (常量标记)
[+08] uint16: count=8 (常量) + uint16: 0
[+0C] float: 骨骼局部 Y_pos - 1.546 = cos(12°)*Y - sin(12°)*Z - 1.546
[+10] float: 骨骼局部 X_pos = X (精确, R²=1.0)
[+14] float: -(骨骼局部 Z_pos - 0.359) = -sin(12°)*Y - cos(12°)*Z + 0.359
[+18] float: 骨骼局部 NZ ≈ 1.099*NZ (近似)
[+1C] float: 骨骼局部 NY ≈ 1.089*NY (近似, R²=0.998)
```

### 意义

1. Part2 是 GPU skinning 的**骨骼局部空间顶点缓冲区**
2. 骨骼旋转角 12° 在 static/dynamic 完全一致 = 绑定姿态的骨骼旋转
3. 位置精确编码 (R²=1.0), 法线近似编码 (R²=0.92~0.998)
4. count=8 可能是顶点格式描述符 (8 个 float 组件?)
5. v0 pre-field = vertex_count 是数据区起始标记

### 可能用途

1. **GPU skinning compact vertex buffer** — 紧凑格式的位置+法线, 用于 vertex shader
2. **Morph target / blend shape reference** — 作为形变参考的基准姿态
3. **LOD 渲染数据** — 用于低精度渲染通道
4. **Bone-space 坐标** — 相对于特定骨骼的局部坐标 (需验证)

### 骨骼位置分析 (失败)

- 骨骼位置存储在 bone entry transform 的 +0x52 偏移处 (局部空间)
- 直接比较 vertex ↔ bone 距离失败: 所有顶点最近于 bone 47 (L Thigh)
- 原因: 骨骼位置是相对父骨骼的局部坐标, 需要构建完整骨骼层级链才能计算世界坐标
- bone entry transform 结构中 t[70:72] = children_count (验证通过)

## §62 Part1 Edge Records + Part3 Submesh Table (新增 0525-G)

### Part1 Edge Record 格式 (flag=0x46)

```
Part1 = [8B header] + [7B × 776 records] (或变长结构)
+0: 2×float (UV boundary coords)
+8: uint16 776 (record count)
+10: uint16 0 (padding)
+12: data 开始
```

**7-byte record**: `[uint16 field1][uint16 vertex_a][uint16 vertex_b][uint8 sep_data]`

**统计 (static, vc=489, fv=589)**:
- 从 +4 偏移: 777 条记录, 588 有效 (75.7%)
- **588 ≈ fv_count = 589**, 但与 FV mesh 零对应关系
- field1 只有 3 个值: 0(231), 1(302), 2(55)
- 与 FV mesh 边仅 4.4% 重叠, 与 Part3 strip 边仅 11.8% 重叠
- 与 FV 三角形 0% 映射 (所有 FV-edge 测试均失败)
- 0x00/0x01 字节占 52.7% — 符合 uint16 小值顶点索引分布

**结论**: Part1 不是 FV mesh 的 edge 数据。可能是内部 mesh 表示
（如碰撞网格或渲染优化用的 winged-edge 结构）。实际修改时可原样保留。

### Part3 Submesh Table

**76B header 后紧跟 submesh_count × 4B 表**:

```
Part3 = [76B header] + [submesh_count × 4B table] + [uint16 index data]
```

~~**Table entry**: `[uint16 field_a][uint16 field_b]`~~

**§62 修正: 无独立 submesh table!**
76B header 后全部是 uint16 index buffer, 99.5% 有效 (<vc)。
前 submesh_count×2 个 uint16 只是被误读为 "table" 的 index buffer 头部。
submesh_count 是 header 元数据, 非数据布局参数。

```
Part3 实际结构:
  [76B header] + [uint16 index buffer (triangle strip + degenerate separators)]
```

| 0x00 | 1194 | 8 | 256 | ~31K (varies) | 7 |
| 0x00 | 1967 | 11 | 256 | ~17K (varies) | 8 |
| 0x6E | 1547 | 10 | 256 | 32512 | 6 |
| 0x75 | 2378 | 16 | 256 | ~21K (varies) | 10 |

**规律:**
- **+0x34 = 256 (0x100) 全局常量** — 所有文件、所有 flag
- **+0x30 随 mesh 复杂度递增** — 可能是 bone influence count 或 vertex stream count
- **+0x28 u16[0] = 256 的倍数** — 可能是 index buffer byte size 或 texture stride
- **+0x28 u16[1] = flag-specific 常数** — 0x46→2, 0x11→3, 0x6E→6, 0x00→7/8, 0x75→10

### Bone +0xB8 分析

| 文件类型 | +0xB8 值 | 含义 |
|---------|----------|------|
| 原始 static MT (50125461) | {1098} | 原生骨骼 parent ID |
| 原始 dynamic MT (50125331) | {1098} | 同 item, 同值 |
| 原始 item 50125671 (flag=0x75) | {1102} | 不同 item, 不同值 |
| res767 (flag=0x00) | {1102}, {1103} | per-item 常量 |
| 注入/weighted 文件 | {953, 原生值} | 953 = 注入骨骼 parent ID |

**结论:** +0xB8 是 per-bone 的 parent reference ID。原始文件所有骨骼共享一个值,注入文件多出 953。

### Edge Data (所有 flag 类型通用格式)

**Header**: `[float f0][float f1][uint32 count]` (12 bytes)

| Flag | f0 | f1 | count | edge_data size |
|------|-----|-----|-------|---------------|
| 0x46 | 0.130462 | 0.421294 | 776 | 21092B |
| 0x11 | 0.018373 | 0.049520 | 1427 | 53265B |
| 0x6E | 0.759902 | 0.383635 | 2605 | 67751B |
| 0x75 | 0.496874 | 0.842069 | 4266 | 131698B |

**记录格式**: 7-byte records `[uint16 a][uint16 b][uint16 c][uint8 sep]` 从 +4 开始

**两段式结构**:
- Section 1 (前 ~1500-4500 条): a,b < vc, 高有效率 (80-96%)
- Section 2 (剩余): 值 >= vc 为主, 有效率 ~3-14%
- 过渡是渐进的, 非突变

**Half-edge 对**: Section 1 中记录成对出现, (b,c) 顺序相反:
```
rec[1]: a=0, b=133, c=488, sep=0x1A
rec[2]: a=2, b=488, c=133, sep=0x7A   ← 相同边, 方向相反
```

**与已知 mesh 的关系**: 0% 匹配 FV 三角形边, 20% 匹配 Part3 三角形边。
说明 edge data 描述的既非 FV mesh 也非 Part3 mesh。

**Header count 与 mesh 关系**:
- flag=0x11: count=1427, FV三角形=309, FV边=880 → count ≈ 1.6× FV边
- flag=0x75: count=4266, FV三角形=882, FV边=2473 → count ≈ 1.7× FV边

**Section 2 sep byte 聚集**: 0x00, 0x3D-0x3F (61-63), 0x80 (128), 0xBD-0xBF (189-191)

### Part3 submesh_count = vc // 10 (§63 突破)

**所有文件验证**: submesh_count = vertex_count // 10, 无一例外。

| vc | submesh_count | vc/sm |
|----|--------------|-------|
| 489 | 48 | 10.2 |
| 850 | 85 | 10.0 |
| 1194 | 119 | 10.0 |
| 1547 | 154 | 10.0 |
| 1967 | 196 | 10.0 |
| 2378 | 237 | 10.0 |

submesh_count 是渲染批次参数, 每 batch 约 10 个顶点。与 strip count 无关 (ratio 0.65-2.06)。

### FV 与 Part3 关系 (§63)

**FV 和 Part3 覆盖 100% 相同的顶点集** — 所有文件都是 FV 和 Part3 各自引用全部 vc 个顶点。
它们是**同一顶点池的不同三角形化**:
- FV: 309 三角形 (flag=0x11), UV-mapped 用于纹理渲染
- Part3: 3049 三角形, triangle strip 编码, 用于 GPU 渲染

Part3 三角形数量 ≈ 10× FV 三角形数, 因为 strip 编码需要重复顶点 (不同纹理坐标)。

### Edge Data 与已知 mesh 的关系

Edge data 描述的是**第三种网格表示**:
- 与 FV 三角形: 0% 重叠
- 与 Part3 三角形: 0% 重叠
- 772/850 顶点被引用 (78 个顶点不在任何边中)
- 最大顶点度 319 → 疑似邻近图而非拓扑网格
- V-E+F = 72 → 非封闭网格
- E (1143) >> 3F/2 (664) → 过多边, 非纯三角形网格
