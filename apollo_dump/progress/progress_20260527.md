### DDynamicActor 构造函数 0x0229AE80 (capstone 反汇编)

```asm
push ebp / mov ebp, esp / push -1           ; SEH frame
push 0x2643026 / mov eax, fs:[0] / push eax
sub esp, 0x18
mov [ebp-0x24], ecx                          ; this = ecx
mov eax, [ebp-0x24]
mov dword ptr [eax], 0x284a9ec               ; *** this->vptr = DDynamicActor::vtable ***
mov dword ptr [ebp-4], 1                     ; SEH state
mov ecx, [ebp-0x24]
call 0x229b1d0                               ; DynamicInit(this)
; DynamicInit 内部:
;   call [vtable+0x10]                        ; 虚函数调用
;   mov [ecx+0x460], ...                      ; 物理参数块1
;   mov [ecx+0x10C], ...                      ; 物理参数块2
mov ecx, [ebp+8]
and ecx, 1 / je 0x229aee7                    ; if (flags & 1)
push edx / call 0x24be4e6 / add esp, 4       ; operator delete(this)
ret 4
```

### DStaticActor 构造函数 0x024C5520 (capstone 反汇编)

```asm
push ebp / mov ebp, esp / push ecx
mov [ebp-4], ecx                             ; this = ecx
mov ecx, [ebp-4]
call 0x2295290                               ; base class ctor
mov eax, [ebp-4]
mov dword ptr [eax], 0x284e0b4               ; *** this->vptr = DStaticActor::vtable ***
mov ecx, [ebp-4]
mov dword ptr [ecx+0x460], 0                 ; 物理块 = NULL
mov dword ptr [edx+8], 3                     ; type = 3 (静态标记?)
mov dword ptr [eax+0x74], 0
ret
```

### 关键差异: DDynamicActor vs DStaticActor

- DDynamicActor: vtable + DynamicInit (物理参数初始化)
- DStaticActor: vtable + base ctor (无物理参数, `[ecx+0x460]=NULL`)
- **两者构造函数均 0 个 E8 直接 caller** — 通过 `operator new` + placement new 间接调用

## §80: CharacterMotion 解析函数 (纠正)

```
CharacterMotion 解析 @ 0x021B42D0 (4153B)
  ├─ cmp [ebp-0x74], 0x28 → jae 0x021B52B1  (循环边界, 非Dynamic/Static)
  ├─ cmp [ebp-0xF4], [ebp-0xC0] → je 0x021B500B  (Type比较, 不等则跳过)
  │   ├─ call 0x021BD290
  │   ├─ call 0x021BD860
  │   ├─ call 0x021C0F00  (顺序执行, 非分支)
  │   ├─ call 0x021C0F40  (顺序执行, 非分支)
  │   └─ call 0x021BFC90, ...
```

**0x021C0F00 和 0x021C0F40 不是 Dynamic/Static 分支**。Dynamic/Static 选择不在 CharacterMotion 解析中，而是通过 `operator new` 分配不同大小对象 + 调用不同构造函数实现。

## §81: 运行时追踪 Actor 对象 (Frida Hook → 崩溃 → Pymem 堆扫描)

### 尝试1: Frida Interceptor.attach (trace_actors.py)
- **结果**: 游戏崩溃
- **原因**: Interceptor.attach 写 .text 段 → 触发 Apollo CRC 检测
- **结论**: ❌ 不可用，任何 .text 写入都会触发

### 尝试2: Frida VEH + DR 硬件断点 (trace_actors_veh.py)
- **原理**: Process.setExceptionHandler() + NtSetContextThread 设置 DR0/DR1
- **结果**: VEH 注册成功，收到 Apollo INT3 异常，但 DR 设置 TypeError
- **问题**: Apollo 自身也用 INT3，VEH 被大量无关异常淹没
- **结论**: ❌ 不可用

### 尝试3: Pymem 堆扫描 (scan_actor_objects.py) ⚠️ 进行中
- **原理**: 扫描所有 committed RW 内存，搜索 vtable 指针
- **vtable 运行时验证**:
  - DDynamicActor vtable @ 0x0284A9EC → 可读，vfunc[0]=0x0229AE80 ✅
  - DStaticActor vtable @ 0x0284E0B4 → 可读，vfunc[0]=0x0236CC20 ✅
  - 两值在 .text 段中分别出现 7 次和 4 次（构造函数和 wrapper 写入）
- **Bug**: VirtualQueryEx 循环返回 0 个区域 → OpenProcess handle 无效
  - `ctypes.windll.kernel32.OpenProcess(0x0410, False, pid)` 可能返回无效句柄
  - 需要改用 pymem 已有的 process handle，或改用 ReadProcessMemory 直接扫描
- **当前状态**: lobby/背包界面扫描 0 对象（符合预期，无角色加载）

### 关键结论
1. Frida Interceptor 不可用（写 .text 触发 CRC）
2. Frida VEH+DR 部分工作但 Apollo INT3 干扰太大
3. Pymem ReadProcessMemory 安全，但 VirtualQueryEx 枚举需修复
4. 大厅/背包界面无 Actor 对象（需要进入比赛才加载角色）
