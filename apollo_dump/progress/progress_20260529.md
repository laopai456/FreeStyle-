## §92: 工厂函数入口修正

IDA 标注的 DStaticActor 工厂地址 0x0236B36B 实际上是函数内部偏移(+0x2B)，**真实入口是 0x0236B340**。

验证: Pymem 读取 0x0236B340 = `55 8b ec 6a ff 68 1e b9 64 02...` (标准 MSVC prologue)，而 0x0236B36B 处是 `e8 b0 a1 15 00` (call 0x24C5520，分配器调用)。

DStaticActor 工厂函数 0x0236B340 流程:
1. 标准prologue + SEH设置
2. 调用 0x24C5520 (内存分配器)
3. 初始化 +0x464 区域
4. 设置 vtable = 0x0284E00C (`mov [eax], 0x0284E00C` at 0x0236B38E)
5. 初始化各字段 (+0x5A4, +0x5CC, +0x5D8 等)

DDynamicActor 工厂函数 0x0229AF00 流程:
1. 标准prologue
2. 调用 0x2295290 (初始化)
3. 设置 vtable = 0x0284A9EC
4. 从全局 0x2A43CA4/C8/CC 读取物理参数写入 +0x460
5. 调用 0x229B130 物理子初始化

## §94: 工厂调用链追溯 (Pymem E8 搜索)

逐层搜索 E8 call 指令找到的所有调用者:

### Static 路径
```
Level 3 (调用者):
  0x0231893D, 0x0231896C → 调用 Level 2
Level 2:
  0x023193E0 (包装函数, 5个字段清零+IAT调用) → 调用 Level 1
Level 1:
  0x0231B930 (DStaticActor创建, 0x1B930-0x1BB4D, ~0x21D字节) → 调用工厂
工厂:
  0x0236B340 (DStaticActor工厂)
```

Static 还有第二条路径:
```
Level 3: 0x0230FD2D, 0x0230FFA5, 0x02310FBB, 0x023111FF (4处)
Level 2: 0x023194E0 (包装函数)
Level 1: 0x0231BDB0 (DStaticActor创建#2)
工厂: 0x0236B340
```

### Dynamic 路径
```
Level 2 (Dynamic#1):
  158 个调用者! 遍布 0x004551xx - 0x021759xx
  → 0x022FA370 (包装) → 0x02299730 (创建) → 0x0229AF00 (工厂)

Level 2 (Dynamic#2):
  10 个调用者: 0x004360CE, 0x004B766B, 0x004B7A9A, 0x016D9107,
  0x01834785, 0x018ECB87, 0x018ECE9E, 0x01E8213A, 0x021755E1, 0x021759BA
  → 0x022F27B0 (创建) → 0x0229AF00 (工厂)
```

### 关键结论
**Static 和 Dynamic 没有共同的分发点 (if/else/switch)。** 它们是完全独立的代码路径，由更上层的业务逻辑决定走哪条路。Dynamic 包装函数有 158 个调用者，说明游戏大量使用 Dynamic actor。

## §95: 工厂地址搜索

搜索工厂地址作为内存中的存储值:
- 0x0236B340 (Static工厂): **0 处** — 不作为函数指针存储
- 0x0229AF00 (Dynamic工厂): 仅 1 处 (0x5197F90F, 堆上，随机数据)
- 0x02299730 (Dynamic创建#1): 2 处 — 0x0284A92C (vtable附近) + 0x09BDB018 (堆)
- 0x0231B930 (Static创建#1): **0 处**
- 0x0231BDB0 (Static创建#2): **0 处**

结论: 创建函数也不作为函数指针存储，全部通过 E8 直接调用。

## §97: CRC 保护分析

### CRC32 函数 0x0243DAB0
```asm
push ebp; mov ebp, esp; or edx, 0xFF
cmp [0x02A994D0], 0    ; 检查查表是否已构建
jnz +5                 ; 已构建则跳过
call 0x0243DA10        ; 构建CRC32查表 (256次展开循环, 多项式0x04C11DB7)
cmp [ebp+0Ch], 0       ; 检查count参数
jbe +0x3B              ; count=0则跳过
; 遍历内存区域计算CRC32...
xor edx, [edx*4+0x02A990D0]  ; 查表
; 最终:
not edx
push edx
call [0x0268F99C]      ; 间接调用(存储/比较CRC值)
ret
```

- 查表位置: 0x02A990D0 (256 entries)
- 已构建标志: 0x02A994D0
- 间接调用目标: [0x0268F99C] (IAT)
- **无直接E8调用者，无函数指针存储**

### CRC32 表构建函数 0x0243DA10
- 256次展开循环，使用多项式 0x04C11DB7
- 结果存储在 0x02A990D0
- 设置标志 0x02A994D0 = 1

### 旧 CRC patch 地址 (已失效)
- 0x001A3C54: 全零，不是有效代码
- 0x001BE222: 不可读
- 这些地址来自旧版本分析，当前版本已不适用

## §100: 待验证方案 — BML + res pak 组合攻击

### 方案
同时做两件事:
1. **Frida hook ReadFile** → 替换 item767 BML 中 50125461 的 mesh 路径:
   `res767\i50125461_MT.smd` → `res767\i50125711_MT.smd` (8个mesh全换)
2. **res767.pak 加 SMD 文件** → 把 res768 中 50125711 的所有 SMD 文件加入 res767.pak

### 原理
- 游戏按 PakNum(767) 读取 res767.pak
- BML 指向 `i50125711_xx.smd` 文件名
- res767.pak 包含这些文件 → 游戏能找到
- 50125711 是动态物品，其 SMD 包含动画数据 → 游戏可能创建 Dynamic actor

### 为什么之前分别试都失败
- 只改 BML(跨pak): res767.pak 里没有 50125711 的文件 → baldness
- 只改 res pak: BML 仍指向 50125461 文件名 → 文件名不匹配 → baldness
- **同时改: BML 指向新文件名 + pak 里有新文件 → 可能成功**

### 实现文件
- BML patch: 已有 `bml_mem_patch.js`，需修改替换规则
- res pak patch: 已有 `res767_patched.pak`（已包含 50125711 的 SMD 数据）
