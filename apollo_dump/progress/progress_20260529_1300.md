## §203: ItemCode定位 (hook_itemcode_hunt/hunt2)

### DynamicCreate1参数中无ItemCode
扫描this对象、栈帧、指针目标 — 均未找到50125461。
ItemCode在DynamicCreate1被调用前已被消费(用于查BML和资源路径)。

### EBP链搜索 (hunt2) — 有发现但不稳定
Call #4, depth=4时:
- `[ebp-0xB8]` → 指针 → `[ptr+0x50]` = INT 50125461
- 返回地址 RVA: 0x1723E84

**但EBP偏移不稳定**: 下一次运行同一位置读到0x616C4665(乱码)。
调用次数也变化(4次→5次)。EBP链遍历导致游戏崩溃。

## §204: 关键发现 — 服务器覆盖补丁

### hook_equip_diag2.py 诊断结果
```
补丁前:  50125461 INT=6,  ASCII=53
替换后:  50125461 INT=0,  ASCII=0   (89处替换)
进房间后: 50125461 INT=8,  ASCII=5   ← 服务器重新写入!
```

**结论: 进房间时服务器下发装备数据, 覆盖了所有内存补丁。**
- 纯内存替换(一次性)不可行
- 必须在服务器写入后、游戏读取前替换

### 时序分析
从trace看, DynamicCreate1各调用时间:
- Call 1-3: 立即(身体/脸/衣服)
- Call 4: +2秒 (发型)
- Call 5: +2秒 (其他)

服务器数据和actor创建之间有**秒级间隔**, 高频定时器(10ms)可覆盖。

## §205: 崩溃分析

| 脚本 | 扫描量 | 崩溃? | 原因 |
|------|--------|-------|------|
| hook_diag (最小hook) | 0 reads | ✅稳定 | 只记数 |
| hook_equip_trace (backtrace) | ~60 reads | ✅稳定 | Thread.backtrace安全 |
| hook_itemcode_hunt2 (EBP+ptr) | ~27000 reads | ❌崩溃 | 大量指针跟踪 |
| hook_equip_replace (EBP walk) | ~10 reads | ❌崩溃 | EBP链不稳定 |
| hook_equip_replace2 (栈扫描) | ~4000 reads | ❌崩溃 | 栈扫描触发保护页 |
| hook_equip_patcher (堆全扫) | 全堆写入 | ✅稳定 | 无运行时hook |

**经验: Frida Interceptor.onEnter中做大量内存读取会崩溃。最小hook是安全的。**
