## 二、计划

### Phase 1: 算法实测验证

**目标**: 确认 f12 算法对实时游戏包的计算结果与实际包一致。

步骤:
1. 用 `apollo_launcher.py` 启动游戏并登录
2. 获取 PID，运行 `py inject_test.py <pid>`
3. 等待 "socket captured"，游戏正常操作产生几个包
4. 输入 `info` / `i` — 查看最后包的 f12_actual vs f12_expected
   - 如果 MATCH → 算法正确，继续
   - 如果 MISMATCH → 需要排查 (可能 seed 不同或 session 影响)
5. 多发几个包，反复 `info` 确认多个 seq 都匹配
6. 特别关注 seq > 194 的情况，验证 L[7] 预测

### Phase 2: 心跳包注入测试

**目标**: 确认服务器接受我们构造的心跳包 (f12 正确)。

步骤:
1. 在正常游戏状态下，输入 `hb` / `heartbeat`
2. 观察结果:
   - `HEARTBEAT OK` + 无异常 → 服务器接受了
   - 连接断开 / 无响应 → f12 可能不对，或心跳包结构有问题
3. 如果心跳被接受:
   - 连续 `hb` 多次，观察是否稳定
   - 用 `loop` 命令做持续心跳模拟

### Phase 3: 递增包注入

**目标**: 注入一个递增 seq 的包，测试服务器处理。

步骤:
1. 正常操作产生一个包
2. 输入 `n` / `next` — 重放最后一个包但 seq+1, f12 正确
3. 观察服务器反应:
   - 如果接受 → 我们可以伪造任意 seq 的包
   - 如果断开 → 可能需要考虑更多因素 (timestamp, payload 校验等)

### Phase 4: 改包注入 (新策略)

**策略**: 拦截 WSASend → 修改加密 payload 的 B[0:1] (b0/b1) → 放行。
seq/f12 由游戏自然生成，无需处理。

步骤:
1. 用 Frida 批量导出全量 ItemCode→b0/b1 映射表
2. 在 WSASend onEnter 中修改 WSABUF 加密 payload
3. 替换 b0:b1 → 服务器侧显示不同外观
4. 后续扩展到衣服、鞋子等其他部位

### Phase 5: 验证周期-8 预测

**目标**: 捕获 seq > 194 的数据，验证 L[7] = 0x07A17A9F。

步骤:
1. 在长时间游戏中自然产生 seq > 194 的包
2. 用 `info` 命令验证 f12 是否匹配
3. 如果 L[7] 不正确:
   - 用 Frida hook send() 前的内存，直接读取 f12 字节
   - 对比算法输出，修正 LEVEL 表

### 26.2 12B Payload 结构模型

```
B[0:1]  = 物品ID (b0+b1, 2B LE)    ← 对应物品的 in-game part ID
B[2:3]  = A8 4A / A8 4B            ← 操作类型标记 (session 内固定)
B[4]    = 属性1 (颜色/槽位等)       ← 可变
B[5]    = B8                        ← 常量, 从发型包到商城包始终不变
B[6:9]  = 属性2-5 (4B)              ← 可变参数
B[10:11]= 72 01 / 77 01            ← 尾部标记 (同类型包固定)
```

8B 短包: B[0:5] 同上, 无 B[6:11] 尾部。

### 26.3 ItemCode → b0/b1 映射突破 ★

**关键发现**：`snapshots/descriptor_scan.json` 中记录了游戏运行时加载的物品描述符，
每个 ItemCode 对应一个内存地址（如 `0x9f7eda8`），描述符中包含 `category_04C`、
`subtype_058` 等字段。b0/b1 (in-game part ID) 也藏在同一描述符的某个偏移中。

**注意**：+0x058 字段在此处被称为 "subtype"，后续分析确认动态发型在此偏移 = 2，静态 = 0。

**实测验证**（商城试用三个静态发型，按点击先后）：

| 点击 | ItemCode | 名称 | subtype | b0 | b1 |
|------|----------|------|---------|----|----|
| 1 | 50125721 | 粉色超级赛亚人3 | 3 | 0x7A | 0xC2 |
| 2 | 50125701 | 紫色护目镜马尾辫 | 1 | 0x7E | 0xBA |
| 3 | 50125691 | 黑金热血青春发型 | 0 | 0x62 | 0x28 |

**验证通过**：用户名→ItemCode→subtype→b0/b1 链完整走通。

### 26.4 b0/b1 动态性分析（已推翻 §26.3 结论）

**§26.3 的 "ItemCode→b0/b1 固定映射" 结论是错误的。**

实测：同一个发型（黑金热血青春）连续试用两次，b0/b1 完全不同：

| 点击 | seq | b0 | b1 |
|------|-----|----|----|
| 黑金#1 | 1851 | 0x70 | 0xE3 |
| 紫色 | 1854 | 0x75 | 0x1D |
| 黑金#2 | 1857 | 0x7A | 0x5C |

连续5次同一发型更证实：

| seq | seq_lo | b0 | b0+seq_lo |
|-----|--------|----|-----------|
| 1880 | 0x58 | 0x53 | 0xAB |
| 1883 | 0x5B | 0x50 | 0xAB |
| 1886 | 0x5E | 0x55 | 0xB3 |
| 1889 | 0x61 | 0x5A | 0xBB |
| 1893 | 0x65 | 0x5E | 0xC3 |

发现：(b0 + seq_lo) 每次换装操作递增 8（周期-8 LEVEL 表模式），说明 b0/b1
是「全局计数器 + 物品属性」的混合值，**不是静态物品 ID**。

结论：
- b0/b1 不在描述符内（扫描 1024 字节无匹配）
- b0/b1 每次发包都变，无法建立 ItemCode→b0/b1 静态映射表
- "改包替换 B[0:1]" 方案不可行

### 26.5 描述符 ItemCode 直接修改 ★★★ 重大突破

**核心发现**：不需要改包，直接修改内存中描述符的 ItemCode 字段即可换装。

原理：
1. 物品描述符在内存中，vtable=`0x027FCEA0`，ItemCode 在 `+0x060`
2. 将描述符 A 的 ItemCode 从 `A` 改为 `B`
3. 用户点击装备 A → 游戏读描述符 → 用 B 的数据自动构造正确包
4. seq/f12/加密全部由游戏自然生成，无需处理
5. 服务器显示 B 的外观

实测验证：
- 黑金(50125691) 的 ItemCode 改为 粉色超赛3(50125721)
- 点击黑金试用 → 显示粉色超赛3外观 ✅

工具：`patch_itemcode.py <pid> <src_ic> <dst_ic>`

优点（相比改包）：
- 零风险：不碰网络包，不存在 seq/f12/seed 问题
- 简单：一个 pymem.write_int 即可
- 通用：理论上适用于所有物品类型

待验证：
- 其他部位（衣服、鞋子等）是否同样有效
- subtype 不匹配时是否出问题
- 描述符是否会被游戏刷新覆盖

### 26.6 描述符刷新问题 & 持久化尝试

**问题**：退出商城重进后 patch 失效。原因：游戏在进商城时一次性创建描述符并缓存数据，
在同一个函数调用（微秒级）内完成，外部轮询来不及。

尝试过的方案：

| 方案 | 方法 | 结果 |
|------|------|------|
| Python 轮询 (1s) | `patch_itemcode.py` watch 模式 | ❌ 太慢，游戏已缓存 |
| Frida 轮询 (50ms) | `patchic` 命令，JS 内 setInterval | ❌ 仍然太慢 |
| Frida hook 拷贝构造函数 | `Interceptor.attach` 0x1C29FE0 | ❌ 游戏崩溃 (SEH 冲突) |
| patch 所有 ItemCode 出现位置 | `patch_source.py` 全量替换 | ❌ "item load failed" (破坏了索引表) |
| 仅 patch 描述符位置 | `patch_source.py` vtable 过滤 | 背包物品无描述符，0 个命中 |

**关键发现**：
- 描述符只在**商城试用**时创建，背包装备不创建描述符
- 背包物品的 ItemCode 存储在非描述符数据结构中（索引表/库存表）
- 拷贝构造函数地址 `0x1C29FE0`（vtable 在 this+0x0C，ItemCode 在 this+0x06C）
- 拷贝构造函数有 SEH，Frida Interceptor 会破坏其异常处理链

### 26.7 描述符结构修正

通过反汇编拷贝构造函数（0x1C29FE0）发现：
- 描述符对象嵌入在一个更大的 outer object 中
- outer object 的 this 指针 = 描述符地址 - 0x0C
- vtable 在 outer+0x0C = descriptor+0x000
- ItemCode 在 outer+0x06C = descriptor+0x060
- 拷贝构造函数通过 `call 0x43b4c0` 复制 ItemCode 子对象（outer+0x06C）

### 26.9 背包装备包抓取 & Call Stack 追踪

**方法**: 在 frida_inject.js 的 WSASend hook 中加入调用栈追踪，对 12B payload
(32B 总长) 装备包自动 dump Thread.backtrace。

新增工具:
- `trace` / `notrace` 命令: 切换调用栈追踪
- `disasm_equip.py`: Frida 反汇编装备调用链中的关键帧

**装备包调用栈** (从底到顶):

```
#14 KiUserCallbackDispatcher → #13 GetPropW → #12-11 CallWindowProcW →
#10 Ordinal2713 → #9 0x21cbc3f → #8 0x21ccf06 → #7 0x1846194 →
#6 0x573793 → #5 0x1b90ed0 → #4 0x1922720 → #3 0x242f7f9 →
#2 0x2438a89 → #1 WSASend → #0 0x244766c
```

**关键帧分析**:

| 帧 | 地址 | 功能 |
|----|------|------|
| #5 | 0x1b90ed0 | 装备函数 — 调用前设置 `ecx=[ebp-0x45D8]`，`call 0x1922720` |
| #4 | 0x1922720 | 包构造函数 — `push 0x17BE` (消息ID), 调用 `0x242f7c0` |
| #6 | 0x573793 | 虚函数调用 `call [edx+0x190]`, 后 `cmp [ebp+0xC], 0x202` |
| #7 | 0x1846194 | 虚函数调用 `call [edx+0x190]`, `ret 0x10` |

**frame #4 (0x1922720) 详细反汇编**:
- 函数入口: 0x1922700 (`push ebp; mov ebp, esp`)
- `push 0x17BE` — 消息 ID (装备操作码)
- `mov ecx, [ebp+0x08]` — 第一个参数 (this)
- `call 0x242f7c0` — 包发送底层

**frame #5 (0x1b90ed0) 上下文**:
- 大栈帧对象 `ecx=[ebp-0x45D8]`
- `call 0x1922720` 前后 `[ecx+0xF00]` 清零
- ItemCode 可能来自 `[ebp-0x45D8]` 对象的某个字段

### 26.10 Frame #5 函数反汇编 & ItemCode 偏移定位

**函数信息**:
- 函数入口: 0x1B8B920 (0x55B0 字节，5025 条指令，巨型 switch-case 消息处理函数)
- 函数有 SEH prologue: `push -1; push handler; mov eax, fs:[0]`
- `0x1b9107f` 是 switch dispatch ( GetMessage → cmp [ebp+0xC], 0x202 )

**关键 switch case** (装备发包):

```
case @ 0x1b90ec5:
  mov ecx, [ebp-0x45d8]      ; this = game state object
  call 0x1922720              ; packet builder
  mov byte [ecx+0xf00], 0    ; clear dirty flag
  jmp switch_dispatch
```

前一个 case (0x1b90e02) 读取 `[edx+0xf00]` dirty flag、`[edx+0xf04]` counter、
`[edx+0x164]+0x18` 三个值做 position/movement 更新。

**包构造函数 0x1922720**:
- 栈帧 0x2020 字节，保存 this 于 `[ebp-0x2020]`
- 写入 4 字节零到 buffer，调用 `0x242f7c0(0x17BE, 4, buffer, &out)`
- 实际 12B payload 由更深层函数从 this 对象读取

**ItemCode 偏移定位 ★★★**: 用 `find_ic_in_obj.py` hook 0x1922720，
扫描 this 对象找到 ItemCode 50122721 (粉色网红主播):

| 偏移 | 上下文 | 说明 |
|------|--------|------|
| **+0xE34** | +0xE30=0x31389d9, +0xE38=0x3015c37, +0xE3C=0x302dd23 | **装备槽位数组** (周围也是 ItemCode) |
| **+0xF0C** | +0xF08=0, +0xF0A=0xc6 | **次要引用** (可能是当前操作对象) |

+0xE30~+0xE3C 是连续的 ItemCode 数组（不同装备部位），偏移 +0xE34
对应发型槽位。

### 26.11 Hook 修改尝试 (失败)

### 26.12 真实装备路径 & call chain

**确认的完整调用链** (frida_inject.js trace + Thread.backtrace):

```
UI 事件 → CallWindowProcW → Ordinal2713 → 0x21cbc3f → 0x1846194
→ 0x573793 (虚函数 call [edx+0x190]) → 0x1b8badd → 0x19247a0 (equip fn)
→ 0x23c6570 (packet builder) → 0x23c66ff → 0x23c68bb → 0x1b989ca
→ 0x1ba787f → 0x242f7c0 (lock+forward) → 0x24387f0 (实际 send) → WSASend
```

**0x19247a0 (equip fn) 分析**:
- 简单函数：将 2 个 float 参数转 int，存入 this+0x188 和 this+0x18C
- 然后调用 0x23c6570 做实际发包
- 参数：(917.0, 484.0, 0) — 可能是装备槽坐标/索引

**0x19247a0 入口状态**:
- ItemCode 50122721 **不在 this 对象中** (3 级指针搜索无果)
- this+0xE34 = 0x02FCD90F (不同的值，不是我们的 ItemCode)
- ItemCode 必须在 0x23c6570 内部通过函数调用获取

### 26.13 内存扫描 ItemCode 定位 (scan_itemcode.py)

### 26.15 0x23c6570 入口追踪

Hook 0x23c6570 (被 0x19247a0 调用的实际装包装包函数)，搜索所有位置：

- 寄存器: eax/edx/ebx/esi/edi → **无 ItemCode**
- this 对象 (+0x400 范围): → **无 ItemCode**
- this→ptr 一级 (+0x100→+0x200): → **无 ItemCode**
- 栈 (+0x100 范围): → **无 ItemCode**

参数: this=0xa38b468, arg0=875.0(float), arg1=502.0(float), arg2=0

**结论**: ItemCode 不作为参数传入 0x23c6570，必须在函数内部通过更深层调用获取。
