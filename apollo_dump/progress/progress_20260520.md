### 1.2 关键发现: field12 依赖 payload 内容

**跨 session 对齐失败:**

```
S4 ↔ S5: 57 共享 b0, 仅 17 个可通过固定偏移匹配 (30%)
S3 ↔ S5: 17 共享 b0, 仅 1 个匹配 (6%)

```

**同一 session 内同 b0 有多个不同 field12 值:**

```
S5 b0=0x00: 5 种不同 sk_xor_t 值
S4 b0=0x00: 3 种不同值
S3 b0=0xD5: 2 种不同值
```

**根因:** seq 1 (dec=`4db8a85447b8a854`) 和 seq 2 (dec=`4db8a85441b8a854`) 同 b0=0x00 但 payload 不同, field12 也不同。

### 1.3 修正后的 field12 模型

```
field12(payload, seq) = F(payload) XOR ((seq & 1) ? U_PARITY : 0)
```

- `F()` 是整个 payload 内容的校验/签名函数, 不仅依赖 b0
- `U_PARITY = 0xDD45AAB8` 仍稳定
- b0 只是 packet type 分类标签, 不决定 field12

### 1.4 T[b0] 表的实际位置

T[b0] 表作为"同 b0 固定 payload 对应的 field12 基础值"仍有参考价值。
163/256 有经验值。缺失条目集中在未触发的游戏操作类型。

### 1.5 结论

**仅靠出站数据无法完全破解 field12。** 入站数据是下一个必需的突破口。

## 四、x64 调用约定 Bug 修复 (04:00-04:30)

### 4.1 发现: NtDeviceIoControlFile 参数映射错误

该 API 有 **10 个参数**, x64 下前 4 个在寄存器 (RCX, RDX, R8, R9), 后 6 个在栈。

**旧代码 (错误):**
```js
this.ioStatus = args[3];   // ← 实际是 ApcContext!
this.outBuf   = args[7];   // ← 实际是 InputBufferLength!
this.outLen   = args[8];   // ← 实际是 OutputBuffer 指针!
```

**新代码 (正确):**
```js
// x64: args[0~3]=寄存器, args[4~9]=栈
this.ioStatus = args[4];   // IoStatusBlock
this.ioctl    = args[5];   // IoControlCode (原本就正确)
this.inBuf    = args[6];   // InputBuffer
this.inLen    = args[7];   // InputBufferLength
this.outBuf   = args[8];   // OutputBuffer
this.outLen   = args[9];   // OutputBufferLength
```

**影响:** 之前所有 `outLen`、`ioBytes`、`outBuf` 读数都是垃圾值。
`outLen=49312224` 等其实是内存地址/栈值。

### 4.2 结构体偏移修正 (x64)

| 结构体 | 字段 | 旧偏移 (错) | 新偏移 (对) |
|--------|------|-----------|-----------|
| `IO_STATUS_BLOCK` | Information | offset 4 | offset 8 |
| `AFD_RECV_INFO` | BufferArray | offset 0 | offset 0 (对) |
| `AFD_RECV_INFO` | BufferCount | offset 4 | offset 8 |
| `AFD_WSABUF` | len | — | offset 0 |
| `AFD_WSABUF` | buf | — | offset 8 |

### 4.3 AFD_RECV 数据路径

修复后的正确流程:
1. `AFD_RECV (0x12017)` → onEnter: 读 `inBuf` (AFD_RECV_INFO) →
   提取 `BufferArray` → `AFD_WSABUF[0].buf` (接收缓冲区), `.len` (缓冲区大小)
2. 保存到 `gRecvPollBufs[handleKey]`
3. `AFD_RECV` 返回 `status=0` 时 → onLeave: 读 `ioStatus` (IO_STATUS_BLOCK) →
   `this.ioStatus.add(8).readU32()` = 实际接收字节数
4. 从 `gRecvPollBufs` 读取缓冲区数据 → 检测 magic `43d58b80`

### 4.4 recv hook 诊断

`recv` 确认真实触发 (HTTP 流量), 但游戏 socket 不走 `recv`。
游戏使用 `AFD_RECV` IOCTL → `AFD_EVENT_SELECT` 事件模型, 绕过上层封装。

## 六、Plan C: Hook E2 入口区分出站/入站 (05:00)

### 6.1 原理

`FreeStyle.exe + 0x2038B80` (E2) 是 field12 计算/验证函数:
- **出站时**: E2 计算 field12 → 加密 → WSASend
- **入站时**: 收到加密包 → 解密 → E2 验证 field12

区分方法:
- E2 被调但 3s 内无对应 WSASend → 入站包
- E2 被调后 WSASend 发同一 seq+f12 → 出站包 (从 PENDING 移除)

### 6.2 frida_e2_inbound.js

| Hook | 目标 | 作用 |
|------|------|------|
| `FreeStyle.exe+0x2038B80` | E2 entry | 读取 WSABUF → 提取 magic/f12/seq/payload → 存入 PENDING |
| `WSASend` | WS2_32 | 收到出站包时从 PENDING 移除 |
| `setInterval(3s)` | 定时器 | 超时 2s 的 PENDING → 判定入站 → send PKT |

### 6.3 集成

`apollo_launcher.py` 新增:
- `JS_E2` → `frida_e2_inbound.js`
- `g_frida_e2` 全局变量
- `on_e2_msg()` 消息处理器 (PKT → packet_queue, e2 → log)
- cleanup 时 `g_frida_e2.unload()`

### 6.4 验证结果: E2 从未被调用

| 文件 | 包数 | E2 命中 | 原因 |
|------|------|---------|------|
| `f12_20260520_104951.csv` | 38 OUT | 0 | 游戏未进入世界 (卡登录屏) |
| `f12_20260520_105502.csv` | 38 OUT | 0 | 同上 (会话完全相同) |

**根因: 登录后的游戏数据不走 WSASend → 走 AFD_SEND 直通。**
两个 38 包会话的 ENC 数据只有 3/38 相同, 但 f12 100% 相同 — 登录握手是确定性的, 仅 nonce/timestamp 变化。

## 十一、AFD_RECV diag 上线 → 根因确认 (12:10-12:20)

### 11.1 S11: AFD_RECV 缓冲区诊断

`f12_20260520_121326.csv` — 38 包 (登录握手, 未进入游戏)。

AFD_RECV onEnter 诊断输出:

```
AFD_RECV diag#1: bufArray=0x554f8a0 bufCount=6 | wsaLen=1024  wsaBuf=0x0 | SKIP: wsaLen/wsaBuf bad
AFD_RECV diag#2: bufArray=0x554f670 bufCount=6 | wsaLen=16384  wsaBuf=0x0 | SKIP: wsaLen/wsaBuf bad
AFD_RECV diag#3: bufArray=0x554f9e0 bufCount=6 | wsaLen=15078  wsaBuf=0x0 | SKIP: wsaLen/wsaBuf bad
AFD_RECV diag#4: bufArray=0x554f9e0 bufCount=6 | wsaLen=16384  wsaBuf=0x0 | SKIP: wsaLen/wsaBuf bad
AFD_RECV diag#5: bufArray=0x554f810 bufCount=6 | wsaLen=32768  wsaBuf=0x0 | SKIP: wsaLen/wsaBuf bad
```

### 11.2 根因: IOCP null-buffer 模式

**所有 AFD_RECV 调用的 `AFD_WSABUF.buf` 都是 `0x0` (空指针)。**

游戏使用 IOCP null-buffer 接收模式:
1. 调用 AFD_RECV 提交接收请求, WSABUF.buf=NULL
2. AFD 返回 STATUS_PENDING (0x103)
3. Kernel 自己分配内存, 完成接收后将数据写入 kernel buffer
4. IOCP completion 触发 → `GetQueuedCompletionStatus` 返回字节数
5. 游戏从 kernel buffer 读取数据 → 全程不经过用户态 WSABUF

**AFD 路线结论: 彻底放弃。** 数据在 kernel 空间, Frida 用户态 hook 根本触不到。

### 19.2 93B 包完整结构破解

对所有 16,073 个包逐字节分析后, 精确划定了加密边界:

```

┌────────────────────────────────────────────────────────────┐
│  OFFSET  │ SIZE │ VALUE              │ 状态              │
├──────────┼──────┼────────────────────┼───────────────────┤
│  B[0:1]  │  2   │ 17 fe              │ 明文, 固定         │
│  B[2:4]  │  2   │ fd 00              │ 明文, 固定         │

│  B[6:8]  │  2   │ 00 00              │ 明文, 固定         │
│  B[8]    │  1   │ 0e                 │ 明文, 固定         │

│  B[11:13]│  3   │ 00 50 00           │ 明文, 固定         │
```

**关键发现:**

- 16,073 包中 **零重复 payload** — 每个包内容都独特

## 二十四、f12 校验和完全破解 (2026-05-20/21 深夜)

### 24.1 分析方法

从 22 个 CSV 文件加载 2,800 个包，跨 session 分析 f12(seq) 模式。

| 分析脚本 | 功能 |
|----------|------|
| `analyze_game_f12.py` | f12 vs seq/d0/b0 关系初步分析 |
| `crack_f12.py` | CRC32/Jenkins/Adler32 等标准校验暴力尝试 |
| `crack_f12_v2.py` | 扩展 CRC 变体 + session-seeded checksums |
| `crack_f12_v3.py` | even→odd XOR 常量验证 (0xDD45AAB8) |
| `crack_f12_v4.py` | f12(seq) 跨 session 一致性验证 |
| `crack_f12_v5.py` | per-CSV 一致性 + 跨 CSV 投票 |
| `crack_f12_v6.py` | 完整查找表 + Δ 分析 + JS 代码生成 |

### 24.2 关键发现

#### 24.2.1 f12 是纯 seq 的确定性函数

- **19/22 个 CSV** 对同一 seq 产生**完全相同**的 f12
- 唯一离群的是 `f12_20260519_181141.csv` (S4: 同 session 不同游戏阶段)
- `seq=1` 在所有 session 中恒为 `0xA848F08D`

```
f12(1) = 0xA848F08D  ← 所有 session 一致
f12(2) = 0xCA6A79B4
f12(3) = 0x172FD30C
...
f12(194) = 0xE11311DA
```

#### 24.2.2 even→odd XOR 恒为 0xDD45AAB8

对所有 97 个 even→odd 转换（seq 为偶数），f12(seq) XOR f12(seq+1) = 0xDD45AAB8。

这证明 f12 **不依赖 payload 内容**，纯粹由 seq 决定。

#### 24.2.3 所有标准校验算法均不匹配

| 算法 | 输入变体 | 结果 |
|------|---------|------|
| CRC32 | 各种 init/final_xor/数据组合 | ❌ |
| Adler32 | 各种 init/数据组合 | ❌ |

## 二十五、f12 算法完全破解 (05-21)

### 25.1 算法本质

f12 是一个 **XOR 二进制计数器**, 以 `0xA848F08D` 为种子:

```
f12(1) = SEED
f12(seq+1) = f12(seq) ^ delta(seq)
```

delta(seq) 由 seq 的二进制结构决定:
- **seq 为偶数**: `delta = 0xDD45AAB8` (固定常量)
- **seq 为奇数**: `delta = g((seq+1)/2)`, 其中 `g(n)` 由 n 的 trailing zeros 决定

### 25.2 g(n) 的 XOR 计数器结构

```
g(n) = BASE_G ^ LEVEL[ctz(n)]
```

- `BASE_G = 0x62228939` (基础 odd delta)
- `ctz(n)` = n 的 trailing zeros 数量
- `LEVEL[k]` 以周期 8 循环

```
LEVEL = [0x00000000, 0x7B2231F3, 0x8D665215, 0x6402E328,
         0xB327F7A3, 0x1881A844, 0x4A21617B, 0x07A17A9F, 0x7460C4CD]
```

周期 8 通过第 5 阶 XOR 差分 (d5=0xA4E4AAD9) 为常数推导而出:
L[9] = L[1] = 0x7B2231F3 ✓
