## 二、522 实测记录

### 27.1 商城描述符 Watch 模式优化 ★

**问题**: `patch_itemcode.py` 原版 1 秒扫描间隔，重进商城时游戏在微秒级创建描述符并缓存，
外部轮询来不及拦截。

**优化**:
1. 扫描间隔从 1s → **50ms** (20x 提升)
2. 新增 `find_descriptor_fast()`: 记住描述符曾经出现的内存区域 (64KB 对齐)，
   后续先在已知区域快速搜索，再全内存扫描
3. 首次全扫描后记住区域，重进商城时快速搜索命中率更高

**结果**: ✅ 重进商城后 patch 效果可以维持。

用法:
```
py patch_itemcode.py <src_itemcode> <dst_itemcode>
```

### 27.2 背包装备路径深入分析

#### 27.2.1 0x23c6570 反汇编 (375 条指令)

函数逻辑:
```
0x23c6570:
  this = ecx (UI 对象, 含描述符 vtable 0x027FCEA0 在 this+0x30)
  if this+4==0 || this+8==0: return

  // 创建迭代器遍历容器
  call 0x23ca7f0([this+4], [this+4]) → iterator

  // 遍历链表
  loop:
    node = iterator.next
    data = [node+4]+8         // data 对象
    if data+0x44==0 || data+0x40==0: continue

    // 虚函数判断类型
    result = data->vtable[0xCC]()
    if result==1:
      wrapper = call 0x24be7c2(data, ...)
      wrapper->vtable[0xD8](this, float, float, float)
    else:
      call 0x23c6740(this, data, float, float, float)  // 大 switch

  // switch on type (0x23c6740):
  // type 1: vtable+0x68
  // type 2: vtable+0x6C
  // type 3: vtable+0x70
  // type 4: vtable+0x74
  // ...
```

**关键**: 函数通过虚函数调用获取数据，ItemCode 不直接出现在参数/寄存器/栈上。

### 27.8 渲染管线复盘 & 方向重评 ★

> 结合 docs/07 动态发型项目、游戏源码、x32dbg 逆向结果重新审视。

#### 27.8.1 之前结论的错误

522 说"客户端渲染完全不依赖内存 ItemCode"，这个表述太绝对。
实际渲染管线 (源码 Character.cpp + DGrpActor.cpp)：

```
LoadItemFile(iItemCode, "i{ItemCode}.bml")
  → 从 item pak 读取 BML (XOR 0xFF 加密的 XML)
  → 解析 <mesh> 路径 (如 "res764\i50125031_MT.smd")
  → 存入 m_ItemCharArray

RefreshItem()
  → 遍历 m_ItemCharArray
  → DGraphicAcquireSMD(SFullName("res764", "i50125031_MT.smd"))
  → AttachSMD(hMesh)
```

渲染**确实依赖 ItemCode**（通过 `"i{ItemCode}.bml"` 文件名查 BML → 取 SMD 路径）。
522 改 +0xE34 等内存位置无效，是因为那些位置不是渲染管线读取的 ItemCode。
模型一旦通过上述管线加载就被缓存，事后改内存中的 ItemCode 当然无效。

#### 27.8.2 离线 pak 方案的限制

docs/07 项目（动态发型功能）已验证：
- BML 完整替换 → 光头（SMD 加载失败）
- 局部替换第 12 块 → 无变化
- **根本原因**: 动态/静态发型是**不同 C++ 类**
  - DDynamicActor (vtable=0x284AA4C) — 有物理初始化 (call 0x229C2D0)
  - DStaticActor (vtable=0x284E114) — 无物理初始化
  - CharacterMotion.bml 由**服务端下发**，不在本地 pak 中
  - 运动类型在对象创建时决定 (工厂函数 0x021C1F00)

**结论**: 离线 pak 修改只能做同类型外观替换（静态↔静态），不能跨动态/静态类型。
且替换后游戏可能因 SMD 加载失败导致光头（跨 pak 引用、文件名大小写等问题）。

#### 27.8.4 推荐方向: Hook DGraphicAcquireSMD ★★★

**原理**: 拦截 SMD 文件加载，把源发型的 mesh 路径重定向为目标发型的 mesh 路径。

**为什么这个方案最可行**:
1. 函数入口 hook — Frida Interceptor 安全，不崩（522 之前都是 hook 函数体内才崩）
2. 在渲染管线的正确位置 — SMD 加载发生在 RefreshItem 中，是模型被缓存之前
3. 不碰网络包 — 不存在 seq/f12/seed 问题
4. 不碰 ItemCode — 直接改 SMD 文件路径参数，绕过 ItemCode 查找
5. 纯视觉替换 — 无论目标发型是静态还是动态，SMD 模型都能加载（物理骨骼不动画但外观可见）

**执行计划**:
1. Hook `DGraphicAcquireSMD` onEnter，读取 SFullName 参数中的 SMD 文件名
2. 如果匹配源发型 SMD 路径（如 `res764\i50122721_MT.smd`），替换为目标路径（如 `res764\i50125721_MT.smd`）
3. 游戏加载目标 SMD → 显示目标外观

**风险**:
- SMD 路径参数可能在栈上/寄存器中，需要确认参数传递方式
- 替换后纹理路径也可能需要同步修改（BML 中的 <texture> 字段）

#### 密钥初始化 (InitializeInstance)
```

2. GenKey(): s_key = rand(); key = s_key ^ s1; key = (key << s3n) | (key >> s4n)

   - 同样抽取 s1_key, s2_key, s3_key, s4_key (服务端密钥)
```

| s3_inc | 3 | 递增步长 |
| s4_inc | 11 | 递增步长 |

| c2_inc | 1 | 递增步长 |
| c3_inc | 9 | 递增步长 |
| c4_inc | 1023 | 递增步长 |

**每包递增 (i++ 后缀)**：每个数据包的密钥 = 上一包的密钥 + N*步长。

#### 数据包加密流程
```
包头8字节:
  原始:  [seq0][seq1][seq2][seq3][cmd0][cmd1][len0][len1]
  加密: [seq3][tag2][len0][tag0][cmd1][tag3][cmd0^s2][tag1]

载荷: 8字节块用 s4_key XOR → 剩余按4/2/1字节递减
  8字节: DWORD[0] ^= s4_key; DWORD[1] ^= s4_key
  4字节: DWORD ^= s4_key
  3字节: WORD ^= s2_key; BYTE ^= s3_key
  2字节: WORD ^= s2_key
  1字节: BYTE ^= s1_key

每包更新: s1_key += s1_inc++; s2_key += s2_inc++; s3_key += s3_inc++; s4_key += s4_inc++
```

#### 数据包解密流程
```
提取: seq = byte[0,3,2,5]; cmd = byte[1,6] ^ c2_key; len = byte[4,7] ^ c2_key
验证: CheckTag(seq) → ++c_tag == (seq ^ cs_key)
载荷: 同加密，用 c4_key/c2_key/c1_key/c3_key 递减处理
更新: c1_key += c1_inc++; c2_key += c2_inc++; c3_key += c3_inc++; c4_key += c4_inc++
```

## 三十六、BML ReadFile 拦截实验 (523 00:27)

### 36.1 假设

基于源码 `Character.cpp` 分析，`LoadItemFile` 通过 `DFileGPack` 从 pak 读取 BML (XOR 0xFF 加密 XML)。
如果 hook `ReadFile` 拦截 BML 内容，替换其中的 `mesh=` 路径，游戏会自然加载目标模型。

### 36.2 实验结果: BML 不走 ReadFile ❌

使用 `hook_bml.py` + `hook_bml.js` hook `kernel32.ReadFile`，诊断模式记录所有小读取。

**发现**:
- ReadFile 仅看到 6144B 二进制 pak 块读取（纹理/索引数据）
- **零次可读文本读取**（无 XML、无文件名、无 mesh 路径）
- 对比：同一 hook 能检测到 SMD 文件的 SSKF 头（二进制）

**结论**: `DFileGPack` 在启动时将 pak 元数据（含 BML）缓存到内存，后续 BML 读取从内存缓存取，不走 ReadFile。只有大型 SMD 模型数据才走 ReadFile。

### 36.3 关键发现: SSKF 是一次性读入 ★

源码 `DGraphicObj.cpp:117-251` 揭示:

```cpp
// AcquireSMD 实现
DFileGPack File;
File.Open(Filename);
BYTE head[4];
File.Read(head, 4);
if (memcmp(head, "SSKF", 4) == 0) {
    File.SeekToBegin();
    DWORD total = File.GetLength();
    BYTE* buf = new BYTE[total];
    File.Read(buf, total);          // ← 一次性读整个 SSKF 文件
    DMemoryReader Reader(buf, total);
    pMesh->Serialize(Reader);
    delete[] buf;
}
```

**SSKF 文件是一次性整体读入内存的**，不是分多个 chunk。这意味着：
- v7 的 hFile 追踪 + 多 chunk 拼接是**不必要的复杂化**
- 只需检测 SSKF header，然后在随后的整文件读取中做数据替换
- 替换窗口是**单次 ReadFile 调用**

### 36.4 对 v10 失败的重新评估

§34.8 记录 v3→v10 全部失败，根因是"SMD 内部引用断裂"。
但源码揭示了真正的失败原因:

| v10 结论 | 源码更正 |
|---------|---------|
| "多线程并发 ReadFile 同一句柄" | SSKF 是一次性读取，不存在并发问题 |
| "SMD body 大小不同" | 整文件读取，大小差异只需在 Allocate 时处理 |
| "纹理/骨骼引用断裂" | `DProgressiveMesh::Serialize` 反序列化全部数据，包括 CtrlPoints 和 Sections，引用在反序列化时建立 |

**可能的失败根因不是"引用断裂"，而是 pak 句柄被污染导致其他文件读取异常。** 如果只替换 SSKF 整文件读取的那一次 ReadFile buffer（不污染句柄的其他读取），可能可以避免崩溃。

### 37.3 新方向: Memory.protect + Interceptor.attach AcquireSMD ★★★

6. 进商城点发型 → 看到 "HOOKED" 消息 = 成功

- ✅ 成功: 后续在 onEnter 中改 SString 参数 → 完成外观替换

## 三十八、Memory.protect + Interceptor 实验结果 (523)

### 38.1 protect + attach + 恢复 rx → 不崩 ✅

| 实验 | 结果 | 日志 |

**结论**: Apollo 确实会定期扫描 `.text` 页属性。`rwx` → 立刻 `rx` 恢复后 Apollo 扫不到。

### 38.2 0x21e25b0 不是 AcquireSMD

**CALL 目标** `FreeStyle+0x21e25b0` 确认是 `E8 XX XX XX XX` 直接调用, 
但函数是**通用文件名/字符串处理函数**, 非 AcquireSMD 专用:

| args[0] 内容 | 调用类型 |
|-------------|---------|
| `PGFN\x02` | .pak 文件 magic 头 |
| `item768.pak` | .pak 文件名 |
| `i50125711` | 物品代码 |
| `u50622961.png` | 纹理文件名 |
| `C:\...\item768.pak` | bp8 中检测到的 .pak 路径 |

**调用次数**: ~80次/秒, 来自多个调用者。

### 38.3 SString inline 字符串位置已确定

```
args[0].readAnsiString(64) → 直接读到文件名/item code
```

**结构**: 短字符串直接存 inline (类似 std::string SSO), 
`ptr0/ptr4/ptr8` 方式读不到有效数据, 必须 `readAnsiString` 直接读。

### 38.4 AcquireSMD return address 过滤失败

**尝试**: 通过 `Memory.readPointer(ESP).sub(base).toInt32()` 过滤 `0x1eec93e` / `0x1eecba7`
**结果**: 零命中 (`acquire_test_20260523_020034.txt` — 0条 ACQUIRE 消息)

**可能原因**:
- `toInt32()` 返回有符号值, 比较逻辑可能有问题
- 或者钩子入口的 ESP 已被 Frida 修改
- 或者 AcquireSMD 的调用不是直接 CALL 而是间接跳转

**下一步**: DIAG 版不打过滤, 前20个调用直接打印真实返回地址。

## 三十九、AcquireSMD 函数入口定位成功 (524)

### 39.1 纯内存扫描定位 (无调试器/Frida)

通过被动内存扫描（搜索已知字符串引用），定位到 AcquireSMD 函数体：

| 地址 | RVA | 说明 |
|------|-----|------|
| **函数体内字符串引用** | **0x01EED020** | push "strSMD is none in AcquireSMD", "SSKF", "RefreshItem.log" — 函数体内部 |
| **函数入口 (CC+SEH prologue)** | **0x01EEC130** | 0x01EED020 上方 0xEF0 字节处，backtrace 分析确认 |

**重要**: 0x01EED020 是函数体内部地址（push 字符串常量），不是函数入口。
真正的函数入口是 **0x01EEC130** (CC padding + SEH prologue)。

§27.9 测试 "0x22EC130 hook 不触发" 是 Apollo 拆除前的结果。
Apollo 拆除后，acquire_hook_log.txt 的 backtrace 证明此函数在 SMD 加载时被调用。

**铁证**（字符串引用在函数体内）:
- `push 0x0284BA0C` → "strSMD is none in AcquireSMD = [%s]"
- `push 0x0281AF2C` → "RefreshItem.log"
- `push 0x0284BA04` → "offline"
- `push 0x0284B9C4` → "SSKF"

### 39.4 AcquireSMD Hook 策略

1. `Interceptor.attach(base.add(0x01EEC130))` — 函数入口 hook (0x01EED020 是函数体内，不是入口)
2. `onEnter`: 遍历 args[0..3], ECX, 栈区域

   - 找到则 `writeUtf8String` 替换为 target item code
3. 替换必须等长 (SString SSO buffer 固定大小)
4. `onLeave`: 日志记录返回值
