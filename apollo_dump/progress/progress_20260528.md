8 个 DS 对象, f46C (idx) = 12~19, 连续编号:

## §90: 描述符 ↔ DS 对象 关联尝试 (失败)

尝试了多种方法建立 DS 对象到 ItemCode 的映射:
1. 描述符 (vtable=0x027FCEA0) 内搜索 DS 对象地址 → 未找到
2. DS 对象内搜索描述符地址 → 未找到
3. 全堆反向指针扫描 → 噪声太大 (idx=18 有 6423 个外部引用)
4. ItemCode 50125461 (美丽梦想发型) 附近 0x1000 搜 DS 指针 → 未找到
5. DS 对象所在内存区域搜 ItemCode → 未找到

结论: ItemCode 和 Actor 对象通过角色对象的**两个独立数组** (装备表 + Actor 表) 用 slot 索引关联, 无直接指针。

## §93: 源码分析 — DD/DS 的真正含义

通过源码分析发现:

**DDynamicActor ≠ 布料/头发物理**, 而是角色运动物理 (重力/碰撞/移动)。
**DStaticActor = 无运动物理的静态对象。**

发型的"动态"效果通过**骨骼驱动的顶点变形**实现 (DProgressiveMesh::UpdateMesh):
- DD 渲染路径: 每帧通过骨骼矩阵变形顶点
- DS 渲染路径: 直接渲染静态 mesh

**核心结论: Actor 类型 (vtable) 不决定视觉效果。** mesh 数据中的骨骼权重才是关键。
- 动态发型 mesh 有骨骼权重 → 变形
- 静态发型 mesh 无骨骼权重 → 不变形
- 改 vtable 不会让没有骨骼权重的 mesh 突然能动

## §95: Pre-launch PAK BML 修改实验

### 95.1 思路

既然运行时 patch ItemCode 只对商城试用有效，背包物品在登录前就决定了 mesh。
尝试直接修改 PAK 中 BML 的 mesh 路径，让游戏加载目标 mesh。

### 95.2 PAK 结构确认

item747.pak (50122931 粉色另类半扎所在 PAK):
- PGFN 格式，24B 头部 + 顺序条目 [16B头][name\0][pad][data] + SUC(16B)
- 条目头: [name_len(4B)][data_offset(4B)][data_size(4B)][end_offset(4B)]
- i50122931.bml: header@0x8D3, name@0x8E3, data@0x8F1, size=1585
- BML XOR 0xFF 加密

### 95.3 关键发现: PAK 已被修改

`dopaklog.ini` 记录:
```
50122931#50124951#2026-03-01 21:04:34
```
`FS改装专用v1.0.exe` 在 2026-03-01 已将 50122931 的 BML mesh 路径从原始 `res765\i50125241` 改为 `res764\i50124951` (全息爱心发型)。

但用户反馈: **仍然没有动态效果**。

### 95.4 BML 修改 (第二次)

将 mesh 路径从 `res764\i50124951` → `res768\i50125711` (紫色超赛，已确认动态有效)
- 同长度替换 (16字节)，直接原地修改 PAK，无需重建
- 16 处 mesh/texture 引用全部替换
- 备份: item747.pak.bak2

### 95.5 核心问题: mesh 文件 (.smd) 位置未知 ★

源码分析确认:
- `DGraphicStore::AcquireSMD` 通过 `DFileGPack` 从 pak 读取 .smd
- .smd 文件在 pak 内部，通过 `SetFilePointerEx` seek + `ReadFile` 读取
- SMD 路径如 `res768\i50125711_MT.smd` 由 BML 提供

**但 cookies 目录 505 个 PAK 全不含 SSKF 数据**。mesh PAK 文件不在 cookies 目录。
可能的存储位置:
1. 游戏本体安装目录 (非 bin/Debug)
2. 运行时从服务器下载缓存到其他路径

**结论**: 只改 BML 路径指向不够，如果目标 mesh PAK 不在游戏可访问路径，会加载失败 (参考 §27.8.2 "BML 完整替换→光头")。需要:
1. 找到 mesh .smd 文件的实际存储位置
2. 或者直接替换 mesh 内容 (把动态 mesh 写到静态 mesh 的位置)
