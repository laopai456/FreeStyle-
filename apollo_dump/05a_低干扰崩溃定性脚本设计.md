# 05a_低干扰崩溃定性脚本设计.md

> 目的：为 `hook_crash_qualify.py` 提供一版**低干扰调试脚本设计方案**。
> 核心目标不是“修”，而是**低污染地定性**：当前崩溃到底更像 Apollo 杀，还是游戏业务崩。

---

## 一、设计目标

这版脚本只回答 3 个问题：

1. **崩在 Apollo 还是游戏主链？**
2. **崩前最后一个成功点是什么？**
3. **崩溃发生在什么阶段：房间前 / 房间 / 进练习场 / 练习场后？**

注意：

- 这版脚本是**定性脚本**，不是完整修复脚本。
- 必须尽量减少对原始现场的污染，避免“为了调试而引入新的崩溃变量”。

---

## 二、总体原则

### 2.1 保留的内容

只保留最小必要观察和替换：

- `Process.setExceptionHandler`
- **唯一必要替换点**：`MSVCR100.dll!sprintf`
- `kernel32.dll!CreateFileA`
- `kernel32.dll!ReadFile`（仅作弱辅助）

### 2.2 去掉的内容

这版必须去掉以下高干扰点：

- `LoadItemFile` patch
- 游戏 `.text` 内部 hook（如 `AcquireSMD` 调用点）
- `DStaticActor` / `DDynamicActor` 构造函数 hook
- Actor 定时采样
- 任何 vtable / 对象字段周期扫描

### 2.3 原因

原脚本最大的问题，不是“看不到东西”，而是**看得太多、改得太多**：

- 一边做业务修改
- 一边 hook 内部代码段
- 一边扫 Actor
- 一边定性 Apollo vs 业务崩

这样会导致：

> 无法确认崩溃到底来自原始替换方案，还是来自“为了诊断额外加进去的 hook / patch / 扫描动作”。

所以低干扰版的原则是：

> **先只做最小替换 + 最小观测，把“谁在杀”定下来，再决定是否进入第二阶段深挖。**

---

## 三、脚本结构设计

---

## 3.1 Python 外壳

Python 侧职责尽量简单：

- attach 进程
- 创建日志文件
- 打印阶段变化和崩溃摘要
- 提供少量交互命令：
  - `status`
  - `phase`
  - `quit`

### 日志文件建议

命名格式：

```text
crash_probe_YYYYMMDD_HHMMSS.txt
```

### Python 侧不要再做的事

- 不要做复杂业务逻辑
- 不要在 Python 侧拼更多修补分支
- 不要加大段交互命令

目标是：

> **Python 只是“壳”，核心判断在 JS 侧，且只做低干扰证据收集。**

---

## 3.2 JS 核心模块划分

JS 侧分成 4 个模块：

1. 轻量阶段机
2. 唯一业务替换点：`sprintf`
3. 文件层观察：`CreateFileA` / `ReadFile`
4. 异常处理器

---

## 四、模块 A：轻量阶段机

### 4.1 状态定义

统一使用以下阶段名：

- `INJECTED`
- `ROOM_LOAD`
- `PRACTICE_LOAD`
- `CRASH`

可选备用：

- `UNKNOWN_POST_ROOM`

不要再混用：

- `ROOM`
- `FIRST_FRAME`
- `PRACTICE_FIRST_FRAME`
- `LOGIN`

因为名字一多，日志回看时容易混乱。

### 4.2 推进规则

低干扰版不要用复杂阶段机，只用最小证据：

#### 规则 1：第一次命中目标 `sprintf` 替换

```text
INJECTED -> ROOM_LOAD
```

#### 规则 2：第二次命中目标 `sprintf` 替换

```text
ROOM_LOAD -> PRACTICE_LOAD
```

### 4.3 为什么这样设计

虽然这仍然是启发式，但比原脚本那种：

- `sprintf` 次数
- `ReadFile` 中 `SSKF` 计数
- 练习场第一帧推断
- 内部函数命中

混在一起要更可控。

`ReadFile` 在低干扰版里只能作为：

> **辅助备注，不直接改 phase。**

---

## 五、模块 B：唯一业务修改点 —— `sprintf`

### 5.1 只做两件事

如果满足：

- 格式串是目标路径格式
- `itemCode == SRC_IC`

则：

1. 替换 `args[2] = DST_IC`
2. 记录一条事件

### 5.2 暂时不要做的事

**默认不要修改 `[ebp-0xD8]`。**

原因：

- 这是第二层修补
- 会污染“原始崩溃定性”
- 会让你无法判断：当前崩溃来自最小替换链，还是来自额外局部变量 patch

建议做成开关：

```python
PATCH_EBP_D8 = False
```

默认关闭。

如果后续需要“低干扰增强版”，再打开。

### 5.3 `sprintf` 记录字段

每次命中只记录：

- 命中次数
- 当前 phase
- `fmt`
- 原始 itemCode
- 新 itemCode
- 调用者返回地址（`returnAddress`）

这样能回答：

- 替换确实发生了没有
- 替换来自哪个调用路径
- 第一次和第二次替换分别发生在哪个阶段

而不是只知道“改过了”。

---

## 六、模块 C：文件层观察

---

### 6.1 Hook `CreateFileA`

用途：

- 确认 BML 是否真实命中
- 确认 Group 路径是否正确
- 确认是否进入 `item768` / `item742` 这类实际目录

### 6.2 记录条件

只在以下情况记录：

- 路径包含目标 `ItemCode`
- 或路径包含 `.bml` 且包含 `item`
- 或路径包含 `res` 且包含目标编号

### 6.3 不做任何修改

低干扰版里：

> `CreateFileA` 只看，不改。

---

### 6.4 Hook `ReadFile`

用途：

- 弱确认是否读到 BML / SSKF
- 作为“崩前最后一个成功点”的辅助证据

### 6.5 只记录的内容

如果检测到：

- 缓冲区开头像 `SSKF`
- 或者像 XML / `<root>`

则记录：

- tag（如 `SSKF` / `XML_ROOT`）
- `bytesRead`
- 当前 phase

### 6.6 明确限制

低干扰版中，`ReadFile` **不能承担强 phase 推断职责**。

不要再像原脚本那样：

```js
if (rfCount > 10) setPhase('PRACTICE_FIRST_FRAME')
```

原因：

- 多线程读文件会污染计数
- 预读/缓存会影响次数
- `ReadFile` 命中不是场景阶段的可靠边界

所以它只能作为：

> **辅助观察，不作强判断。**

---

## 七、模块 D：异常处理器（最核心）

### 7.1 必须保留的字段

低干扰版异常报告中必须包含：

- `phase`
- `module`
- `eip`
- `rva`
- `instruction bytes (16)`
- `details.type`
- `details.memory.address`
- `details.memory.operation`
- `eax/ecx/edx/ebx/esp/ebp`
- `backtrace`（前 8~10 帧）
- `last_events`（最近 8 条关键事件）

### 7.2 verdict 必须更保守

不要像原脚本那样直接输出：

- `APOLLO_KILL`
- `BUSINESS_CRASH`

低干扰版建议改为：

- `STRONG_APOLLO_SIGNAL`
- `WEAK_APOLLO_SIGNAL`
- `STRONG_BUSINESS_SIGNAL`
- `WEAK_BUSINESS_SIGNAL`
- `UNKNOWN`

因为原始证据很多时候并不足以“一步定案”。

---

### 7.3 verdict 规则建议

#### Strong Apollo Signal
同时满足：

- 崩溃模块落在 Apollo 地址段
- 且指令字节中有 `0xCC`

或：

- backtrace 前几帧明显位于 Apollo 模块
- 且异常类型更像主动中断/处置

#### Strong Business Signal
同时满足：

- 崩溃模块落在游戏主模块
- `details.memory.address` 很小（如 `< 0x1000`）或明显无效
- backtrace 主体也在游戏模块

#### Weak Apollo Signal
- 模块在 Apollo 区，但证据不够硬
- 或 backtrace 含 Apollo，但崩点不强

#### Weak Business Signal
- 模块在游戏区，但不一定是典型空指针
- 或只是一般 access violation

#### Unknown
- 证据混杂
- 模块归属不明确
- 无法仅凭当前日志判断

## 八、事件缓存设计

原脚本只存 checkpoint 名称，这不够。

低干扰版建议统一改成：

```js
var events = [];

function pushEvent(kind, data) {
    events.push({
        kind: kind,
        phase: phase,
        ts: Date.now(),
        data: data
    });
    if (events.length > 20) events.shift();
}
```

### 8.1 需要缓存的事件

至少缓存：

- `sprintf_patch`
- `create_file`
- `read_file_tag`
- `phase_change`

### 8.2 崩溃时输出

异常报告中带出最近 8 条事件：

- 不是只输出名字
- 而是输出完整简化 payload

这样你回看时才能知道：

- 最后一次替换发生在哪
- 最后一次打开的 BML 路径是什么
- 最后一次读到的是 `SSKF` 还是 XML

---

## 九、允许与禁止的 hook 边界

### 9.1 允许的 hook

低干扰版允许：

- `MSVCR100.dll!sprintf`
- `kernel32.dll!CreateFileA`
- `kernel32.dll!ReadFile`
- `Process.setExceptionHandler`

### 9.2 禁止的 hook

低干扰版禁止：

- 游戏 `.text` 内任意调用点
- 角色构造函数
- vtable 附近位置
- 任何 Actor 字段周期采样

理由：

> 这些行为虽然“信息量大”，但也最容易把现场搞脏，甚至直接引入 Apollo 干扰或业务副作用。

## 十、Python 配置项建议

Python 顶部建议只保留少量开关：

```python
SRC_IC = 50125461
DST_IC = 50125711

PATCH_EBP_D8 = False
ENABLE_READFILE = True
ENABLE_VERBOSE_FILELOG = False
```

### 10.1 含义说明

- `PATCH_EBP_D8`
  - 默认 `False`
  - 低干扰版不改局部变量

- `ENABLE_READFILE`
  - 是否启用 `ReadFile` 弱辅助观察

- `ENABLE_VERBOSE_FILELOG`
  - 是否详细打印文件层命中日志
  - 默认建议关闭，避免日志过多淹没关键信息

---

## 十一、日志输出格式建议

### 11.1 普通运行期输出

只打最关键几类：

```text
[phase] INJECTED -> ROOM_LOAD
[sprintf] #1 fmt=... 50125461->50125711 ret=...
[file] Resource\item768\i50125711.bml
[read] tag=SSKF bytes=269824
[crash] verdict=... module=... eip=...
```

### 11.2 崩溃总结块

建议统一成：

```text
===== CRASH SUMMARY =====
phase: PRACTICE_LOAD
verdict: STRONG_BUSINESS_SIGNAL
module: FreeStyle.exe
eip: 0x01ABCDEF
rva: 0x...
type: access-violation
instruction: ...
memory: read @ 0x00000000
registers: eax=..., ecx=...
recent events:
  [ROOM_LOAD] sprintf_patch fmt=...
  [ROOM_LOAD] create_file Resource\item768\i50125711.bml
  [PRACTICE_LOAD] sprintf_patch fmt=...
  [PRACTICE_LOAD] read_file tag=SSKF bytes=34816
backtrace:
  FreeStyle.exe+...
  FreeStyle.exe+...
=========================
```

这样一眼就能知道：

- 崩溃在哪个阶段
- 更像 Apollo 还是业务崩
- 崩前最后发生了什么

## 十二、推荐调试使用顺序

### 第一步：先跑低干扰版

目的：

- 不证明怎么修
- 不深入对象细节
- 只做第一层定性：Apollo 还是业务崩

### 第二步：如果低干扰版显示强业务崩

再上第二阶段脚本，去观察：

- Actor 类型
- motion type
- 物理字段
- 对象生命周期

### 第三步：如果低干扰版显示强 Apollo 信号

再单独做 Apollo 干扰排查版：

- 不和对象采样脚本混跑
- 不和业务层 patch 同时进行

---

## 十三、为什么这版比原脚本更合理

这版方案的价值，在于它解决了原脚本最严重的 5 个问题：

### 13.1 不再双重修改业务参数

去掉 `LoadItemFile` 二次 patch，避免把“替换链本身”和“调试补丁副作用”混在一起。

### 13.2 不再 hook 游戏 `.text`

减少引发 Apollo 检测、页属性变化、副作用崩溃的概率。

### 13.3 不再定时扫 Actor

避免读到已释放对象、半初始化对象、噪音对象，降低误判。

### 13.4 verdict 更保守

不再轻易下“已经定案”的判断，而是按证据强弱给信号等级。

### 13.5 事件缓存更完整

崩溃前上下文更清楚，不再只有模糊的 checkpoint 名称。

---

## 十四、最终建议

低干扰版的核心不是“信息最多”，而是：

> **用最少的 hook、最少的修改，先把“谁在杀”这件事定性。**

只有在这一步完成以后，才值得进入第二阶段：

- 观察 Actor 状态
- 跟对象生命周期
- 研究动态初始化缺口

否则会一直陷在：

- 调试脚本本身制造变量
- 业务 patch 和诊断 patch 混在一起
- Apollo 干扰与业务崩互相污染

---

## 十五、一句话版本

> **先做一个只含 `sprintf + CreateFileA + ReadFile + ExceptionHandler` 的低干扰定性脚本；不再额外 patch `LoadItemFile`，不再 hook 游戏 `.text`，不再扫 Actor。先定性 Apollo vs 业务崩，再决定是否进入第二阶段对象状态调试。**
