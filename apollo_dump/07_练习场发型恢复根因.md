# 07_练习场发型恢复根因与调试教训

> 2026-06-01 补充。基于 sprintf_replace 两次实测日志，确认练习场发型恢复的真正原因。
> 同时汇总所有与 AcquireSMD hook 相关的失败经验。

---

## 一、核心发现：练习场绕过 sprintf 路径

### 1.1 实验证据

使用 `hook_sprintf_replace.py`（含 ReadFile + AcquireSMD 诊断 hook）实测，日志如下：

**房间阶段（01:00:30）**：
- sprintf #8: IC=50125461 → **替换为 50125711** ✅
- AcquireSMD #10: `res768\i50125711_FN.smd` ✅（使用替换后的 ItemCode）

**进练习场过渡（01:00:35，4 轮加载）**：
- sprintf #18,#28,#38: 全部替换成功 ✅
- AcquireSMD #22,#34,#46: 全部显示 `res768\i50125711_FN.smd` ✅

**练习场实际运行（01:00:47）**：
- **零 sprintf 调用**
- AcquireSMD #58: `res767\`**`i50125461_FN.smd`** ← **原始 ItemCode！**

### 1.2 结论

> **练习场阶段不调用 sprintf，直接通过 AcquireSMD 加载原始 ItemCode 的 SMD 资源。**
> sprintf 替换在过渡阶段虽然生效，但练习场渲染用的是绕过 sprintf 的第二条路径。

### 1.3 数据流（更新版）

```
房间/过渡阶段:
  sprintf("customize\item\i%d.xml", 50125461)  ← hook 可替换 ✅
    → LoadItemFile → BML → AcquireSMD(50125711) ✅

练习场实际渲染:
  AcquireSMD("res767\i50125461_FN.smd")  ← 绕过 sprintf，直接加载原始资源 ❌
  无 sprintf 调用，hook 完全无效
```

## 二、SRC_IC 没有被清零

### 2.1 内存扫描对比

使用 `scan_itemcode_diff.py` 做两次内存扫描：

| 阶段 | 含 SRC_IC(50125461) 的地址数 |
|------|------|
| 房间 (scan1) | 12 |
| 练习场 (scan2) | 13（多1个新地址） |

**12 个地址全部还在，SRC_IC 没有被清零。**

### 2.2 推翻之前的假设

doc 05 §11.2 中观察到 `+0x7DC = 0`，推断"练习场把 ItemCode 清零"。实际情况：
- SRC_IC 仍然存在于内存中
- `+0x7DC` 不是练习场渲染的数据来源
- 练习场根本不通过 sprintf/ItemCode 查找路径加载发型

## 三、hook_sprintf_replace.py 导致崩溃的原因

### 3.1 现象

使用 `hook_sprintf_replace.py` 进练习场必定崩溃（无论 DST_IC 是 50125711 还是 50122451）。

### 3.2 崩溃原因

脚本包含两个**游戏 .text 地址**上的 hook：

| hook | 地址 | 安全性 |
|------|------|--------|
| AcquireSMD 调用者 | `0x01EEBA30`（游戏代码） | ❌ 危险 |
| ReadFile | `kernel32.dll`（系统 DLL） | ✅ 安全 |
| sprintf | `MSVCR100.dll`（系统 DLL） | ✅ 安全 |

- AcquireSMD hook 在房间阶段触发次数少（~12次），暂时不崩
- 练习场阶段资源加载量暴增（ReadFile 从 #122 到 #270），AcquireSMD 频繁触发
- **高频触发游戏 .text 段 hook → Apollo 检测到页属性变化 → 杀进程**

### 3.3 之前"不崩"的脚本

doc 06 中不崩的配置用的是 `hook_crash_qualify.py`，它**不包含 AcquireSMD hook**。

### 3.4 教训

> **不要在游戏 .text 段上加诊断 hook。即使短时间不崩，练习场等资源密集场景必定触发 Apollo。**

## 四、AcquireSMD hook 相关全部失败经验汇总

以下来自 `failed_approaches.md` + 本次实测：

| # | 方案 | 结果 | 根因 |
|---|------|------|------|
| 2.1 | AcquireSMD backtrace 定位 (v1) | ❌ | 选错函数，是文件I/O层不是AcquireSMD |
| 2.2 | 逐级 probe 候选函数 args | ❌ | SString 在 AcquireSMD 层被消费为 pack entry index |
| 2.3 | 栈扫描找 .smd 文件名 | ❌ | 文件句柄已用原始文件名打开 |
| 2.4 | 格式字符串 xref 搜索 | ❌ | 格式字符串是死代码/调试残留 |
| 2.11 | Hook 0x22EC130 (Apollo拆除前) | ❌ | 不触发（Apollo阻止） |
| 2.12 | 返回地址过滤 | ❌ | toInt32 有符号比较问题 |
| 2.13 | 热函数 0x21e25b0 hook | ❌ | 80次/秒，hook开销阻塞主循环 |
| **本次** | **AcquireSMD hook @ 0x01EEBA30** | **❌ 崩** | **练习场高频触发→Apollo检测页属性** |
| 1.1 | ReadFile SMD 数据替换 (8轮) | ❌ | 多线程共用pak句柄→数据污染 |
| 2.9 | Stalker 全线程 diff | ❌ | 游戏卡死 |
| 2.10 | Stalker 主线程 diff | ❌ | 超时崩溃 |

### 结论

> **AcquireSMD 相关的所有 hook 方案全部失败。**
> 原因：
> 1. Apollo 保护游戏 .text 段，hook 即被检测
> 2. AcquireSMD 是高频函数，即使 Apollo 停止也会因 hook 开销导致问题
> 3. SMD 资源路径在 AcquireSMD 调用前已被消费为 pack entry index，无法从外部替换
>
> **AcquireSMD hook 替换方向已确认是死路，不应再尝试。**

## 五、当前状态总结

### 5.1 已确认的事实

| 事实 | 证据 |
|------|------|
| sprintf 替换在房间阶段完全有效 | AcquireSMD 加载 i50125711 的 SMD |
| sprintf 替换在练习场过渡阶段也有效 | 4 轮 sprintf 全部替换成功 |
| **练习场渲染绕过 sprintf** | 零 sprintf 调用，直接 AcquireSMD 原始 ItemCode |
| SRC_IC 内存中未被清零 | 12/12 地址在练习场仍含 50125461 |
| 游戏 .text hook 在练习场必崩 | AcquireSMD/ReadFile 高频触发→Apollo 检测 |
| 同类型替换（静态→静态）安全 | doc 04 §七方案1 |
| 跨类型替换（静态→动态）练习场崩 | 骨骼不兼容，doc 04 §一 |

### 5.3 下一步推进方向（2026-06-02 继续）

#### 已确认结论

| 测试 | 房间 | 练习场 | 结论 |
|------|------|--------|------|
| 静态→动态 | ✅ 生效 | ❌ 恢复默认 | 练习场绕过 sprintf |
| 静态→静态 | ✅ 生效 | ❌ 恢复默认 | **问题不是跨类型，是练习场有第二条路径** |

#### 练习场数据来源推测

1. **服务器下发的装备数据** — 进房间时服务器发送角色装备，练习场直接用这份缓存
2. **内存中的预加载缓存** — 房间阶段已加载的资源被缓存，练习场直接引用原始 ItemCode

#### 可能的切入点

| 优先级 | 方向 | 方法 |
|:---:|------|------|
| P0 | **找练习场专用加载函数** | hook 练习场场景切换，追踪角色加载调用链 |
| P1 | **找装备数据缓存位置** | 内存扫描 ItemCode，看练习场阶段哪个地址被读取 |
| P2 | **hook 角色创建函数** | 练习场创建角色时，追踪发型参数来源 |

#### 注意事项

- 不要在游戏 .text 段加 hook（Apollo 检测）
- 系统DLL hook（sprintf/ReadFile）安全
- 需要找练习场专用的数据入口点
