## 一、加载管线（源码级验证 ✅）

```
SetCharacterFeature(itemCode)
  → LoadItemFile → BML (XOR 0xFF → XML) → <mesh>/<texture> 路径
  → DGraphicAcquireSMD(path) → 读 SSKF magic → pMesh->Serialize()
  → BuildVertexBuffer() → 渲染
```

- **零校验**：不检查 bone count、vertex count、版本号
- **骨骼按名称匹配**：`BuildBoneIndex` 逐名比对，失败 → `aBoneMap[i]=-1` → 角色不可见
- 同角色类型（男→男 / 女→女）骨骼名一致，100% 安全
- GPack PAK：无加密、无压缩、无校验和
- BML从PAK加载后缓存到内存，不走ReadFile（拦截ReadFile无效）

## 二、动态/静态发型机制（已确认 ✅）

- **CharacterMotion.bml** 由**服务器下发**，每个 ItemCode 有 `Type` 字段
- Type → 工厂函数 → 创建 **DDynamicActor**（有物理初始化）或 **DStaticActor**（无物理）
- PAK 中动态发型的 SMD 文件**没有物理骨骼**（59 骨 vs 完整版 75 骨）
- 物理效果完全由引擎运行时驱动，SMD/BML 文件不包含物理参数

**决策链**：
```
CharacterMotion.bml (服务器) → 解析函数 0x21B46E0 → Type 字段
→ 工厂 0x021C1F00 → DDynamicActor 或 DStaticActor
```

**Key finding**：Actor类型在创建时决定，换发型后不回退（DS→DD不转换）

## 五、PAK / BML / SMD 文件格式

### 5.1 PAK
- 魔数：PGFN（或NFGP）
- 无加密、无压缩、无校验和
- 头大小不统一（20B vs 16B两种），PGFNPak解析可处理
- 条目布局：[header(16-20B)][name\0][padding][data]

### 5.2 BML
- XOR 0xFF 加密的XML文件
- 结构：`<root><channel>` → `<character type="N"><object><type>normal</type><mesh>..</mesh><texture>..</texture></object></character>`
- item PAK中BML有两种存放方式：
  - **独立条目**：PGFNPak可解析到独立.bml文件
  - **间隙数据**：BML数据位于PAK条目之间的gap中，需用XOR搜索提取
- 不包含物理参数，只包含模型/纹理引用

### 5.3 SMD / SSKF
- 魔数：SSKF（4B）
- Header(328B) + Bone Table(可变) + Material(149B) + Vertex/Index Data
- 版本号 v1（源码 DProgressiveMesh::Serialize 确认）
- 骨骼：SName Name + SName Group + ParentIndex + Children + Depth + Flags + Pos + Quat + Scale

### 5.4 itemshop.json 物品数据库
- **路径**: `C:\Users\w\.claude\projects\C--Users-w-Documents-GitHub-cc\memory\itemshop.json` (~2.9MB)
- **用途**: 物品代码 → 名称/PAK编号/类别/描述 映射查询
- **查询方式**:
  - `python apollo_dump\search_itemshop.py` — 交互式搜索
  - `from item_lookup import name, pak, info, search` — 代码中import
- **示例数据字段**: `name`, `pak`, `category`, `desc`, `attributes` 等

### 6.2.1 静态BML替换 — 当前唯一成功方案 ⭐

> **已成功渲染紫色超赛发型！** 不进练习场就不会崩，唯一阻塞是 Apollo ~10秒杀进程。

**原理**：
- 把紫色超赛的BML（XOR 0xFF编码）放到 `Resource\item\i50125711.bml`
- 通过 sprintf hook (MSVCR100.dll) 把美丽梦想(50125461)的ItemCode替换为50125711
- 游戏 SeekFile 先查松散文件 → 命中 → 加载超赛BML → 正常渲染

**结果**：
| 场景 | 渲染 | 崩溃 | 说明 |
|------|------|------|------|
| 进房间（不进练习场） | ✅ | 不崩 | **完美工作** |
| 进练习场 | ✅ | 💥 崩溃 | 练习场第二次加载时动画系统访问不存在的物理骨骼 |
| Apollo检测 | — | ❌ 杀 | ~10-12秒后 access-violation 或 TerminateProcess |

**根因（练习场崩溃）**：美丽梦想(50125461)是DStaticActor，紫色超赛SMD引用的是动态骨骼。房间内只加载一次mesh → 勉强渲染；练习场重新初始化动画系统 → 访问不存在骨骼 → 崩溃。

**Apollo检测特征**：
- 必须设置 `Process.setExceptionHandler`（return false）传递Apollo的INT3断点
- 崩溃类型：access-violation at 0x695033bc 或 TerminateProcess
- Apollo使用单步追踪（type=system at 0x770a90e2 ntdll）

**脚本位置**：`apollo_dump\hook_diag_crash.py`

### 6.2.2 Apollo 全面压制方案 — apollo_killer.py ⭐⭐⭐

> **双重防护**：页属性欺骗 + 堵死杀进程的全部4条通路。已实现待测试。

**分析结论**：
- ApolloCT.dll 0保护线程（x64dbg_enabler.js diagnose 确认）→ 杀进程来自 FreeStyle.exe 内建的 ApolloShell (L2)
- 之前的页属性欺骗（NtQueryVirtualMemory → 返回 PAGE_EXECUTE_READ）可能被 syscall 级绕过
- 既然无法100%防检测，就**堵死杀进程通路** — Apollo 能检测到我们但杀不了我们

**防护层级**：

| 层级 | Hook点 | 作用 |
|------|--------|------|
| L1 防止检测 | NtQueryVirtualMemory → 欺骗返回 ER | 常规页扫描 |
| L1 防止检测 | NtProtectVirtualMemory → 拦截修改 | 阻止Apollo自己改页属性 |
| L1 防止检测 | VirtualQuery (kernel32) → 欺骗返回 ER | 防御纵深 |
| L2 防止崩溃 | Process.setExceptionHandler → 传递Apollo异常 | INT3/单步追踪不触发杀 |
| **L3 堵死杀通路** | **NtTerminateProcess** → block | **syscall级别最终防线** |
| **L3 堵死杀通路** | **TerminateProcess** → block | **kernel32级别** |
| **L3 堵死杀通路** | **RtlExitUserProcess** → block | **ntdll内部退出** |
| **L3 堵死杀通路** | **ExitProcess** → block | **最上层退出** |
| 额外 | NtRaiseHardError → suppress | 系统错误弹窗 |
| 发型替换 | sprintf + [ebp-0xD8] → 50125711 | 发型ItemCode替换 |
| 发型替换 | LoadItemFile → ItemCode替换 | 第二条代码路径 |

### 6.2.3 rwx→hook→立即恢复rx 方案 (备用)

> 避免Apollo检测页属性变化的方案，已验证不崩。已整合进 apollo_killer.py 的L1防护层。

**原理**：
- Frida Interceptor.attach 会将目标页改为 RWX（可写可执行）
- Apollo 定期扫描 .text 页属性，发现 RX→RWX 变化即杀进程
- attach 完成后**立即**用 VirtualProtect 将页属性恢复为 RX
- Trampoline 仍有效（Frida 的 inline hook 机制不依赖页保持 RWX）

**限制**：只适用于**冷函数**，但页属性欺骗已覆盖此场景。

### 6.3 动态/静态不兼容的根本原因
- 美丽梦想发型(50125461)：**DStaticActor**，无物理骨骼
- 紫色超赛发型(50125711)：**DDynamicActor**（但引擎判断仍创建DStaticActor）
- 静态Actor的mesh初始化流程不同，动态SMD中的额外数据被忽略
- **解决方向**：在渲染层兼容（而非文件/加载层）

### ## 关键发现


1. **Group=0x320(800)是角色组号，不是item PAK号**。所有item共享同一个Group，由角色c800决定
2. **BML查找路径固定**: Group永远是"item"(字符串)，不是整数800
3. **0x1ACDF33(DFileGPack::Open)不可用Frida hook** — 会崩溃，无论入口hook还是中间hook
4. **BML在item768.pak但游戏只查item.pak** → 找不到 → 光头
5. **修复方案**: 松散文件 `Resource\item\i50125711.bml`，SeekFile优先查松散文件
6. **BML需XOR编码**: LoadBinaryXML总是做XOR 0xFF解码，松散文件必须是编码版
