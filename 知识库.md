## 一、核心结论

### 1.1 物理/动态发型机制（已确认）

**动态/静态是运行时 C++ 类决策，不是文件层面属性。**

- `CharacterMotion.bml` 由**服务器下发**，每个 ItemCode 有 `Type` 字段
- Type → 工厂函数 → 创建 `DDynamicActor`（有物理初始化）或 `DStaticActor`（无物理）
- PAK 中动态发型的 SMD 文件也**没有物理骨骼**（59 骨 vs 完整版 75 骨）
- 物理效果完全由引擎运行时驱动，SMD/BML 文件不包含物理参数

**决策链**：
```
CharacterMotion.bml (服务器) → 解析函数 0x21B46E0 → Type 字段
→ 工厂 0x021C1F00 → DDynamicActor 或 DStaticActor
```

### 1.2 加载管线（源码级验证，95%+ 确信度）

```
SetCharacterFeature(itemCode)
  → LoadItemFile → BML (XOR 0xFF → XML) → <mesh> 路径
  → DGraphicAcquireSMD(path) → 读 SSKF magic → pMesh->Serialize()
  → BuildVertexBuffer() → 渲染
```

- **零校验**：不检查 bone count、vertex count、版本号
- **骨骼按名称匹配**：`BuildBoneIndex` 逐名比对，失败 → `aBoneMap[i]=-1` → 角色不可见
- 同角色类型（男→男/女→女）骨骼名一致，100% 安全
- GPack PAK：无加密、无压缩、无校验和

### 1.3 Apollo 防御（已分类）

| 层 | 组件 | 状态 |
|----|------|------|
| L0 | Apollo.sys 内核驱动 | `sc stop ApolloProtect` 停止，文件已重命名 |
| L1 | ApolloCT.dll | **0 保护线程，已死**（CRC patch 点 RVA 0x1A3C54 + 0x1BE222） |
| L2 | FreeStyle.exe 内建反调试 | 阻止调试器附加，Frida 不触发 |
| L3 | .text 运行时解密 | 磁盘上加密（熵 7.23），运行时 Pymem 可 dump |

### 1.4 协议架构

| 通道 | 协议 | 加密 | 状态 |
|------|------|------|------|
| TCP 控制层 | TLS → 61.170.80.79:443 | magic `43d58b80` + XOR key `4db8a854` + f12 校验 | f12 完全破解 |
| UDP 游戏数据 | → 223.166.22.186:18417 | AES 内置实现（S-box 在 .rdata 0x2442ef8） | 密钥未破 |

## 二、SSKF/SMD 文件格式

### 2.1 文件布局（v1，源码 DProgressiveMesh::Serialize 确认）

```
Header (328B)
  +0x00: "SSKF" magic (4B)
  +0x04: version=1 (4B)
  +0x08: Mesh name (null-term ASCII + 4B align)
  +骨骼表标记: 0x75 + uint32 count (实际骨骼数+1)

Bone Table (每骨骼可变长)
  SName Name + SName Group + ParentIndex(4B) + NumChildren(4B)
  + Depth(4B) + Flags(4B) + Pos(12B) + Quat(16B) + Scale(12B, v≤1)

Material Section (149B 固定)
  +0x00: uint32=1 (材质数量)
  +0x04: 材质名 (64B)
  +0x44: flag byte ('s'=static/'g'=dynamic) + padding
  +0x50: 纹理路径 (64B)
  +0x90: mesh_flag (0x46=static/0x6E=dynamic/0x11/0x75/0x00 多种)
  +0x91: uint16 vertex_count
  +0x93: uint16 marker

数据区
  Position channel (vc × 12B)
  Normal channel (vc × 12B)
  FV UV+Idx (每面顶点 10B: float U + float V + uint16 idx)
  Part1 = Faces (SSkinTriangle, 7B: byte MaterialIndex + uint16 WedgeIndex[3])
  Part2 = Influences (SBoneInfluence, 32B: float Weight + int BoneIndex + SVector LocalPos + SVector LocalNormal)
  Part3 = DProgressiveMesh 扩展 (LOD + Sections + Blends 等)
```

### 2.2 关键纠正

- 版本号：源码写 v0，实际文件全部 v1（SBone 有 Scale 字段 +12B）
- SName 格式：不是 FString（长度前缀），是 null-term ASCII + 4B 对齐
- 早期 vtable 地址 0x0284AA4C/0x0284E114 是 vftable 子表，非主 vtable

### 2.3 BML 文件格式

- XOR 0xFF 加密的 XML
- 结构：`<root>` → `<channel>` + `<character type="N">` → `<object>` → `<type>/<mesh>/<texture>`
- mesh type: normal=0, fskin=2, uskin=3, lskin=4
- **不包含物理参数**

### 2.4 PAK 格式 (GPACK/NFGP)

```
文件头: "NFGP" + version + dwCryptMethod(=0) + 元素目录
元素: {uint32 nameSize, uint32 pos, uint32 size, uint32 next, char name[]}
文件尾 SUC: "SUCI" + version + PackHeaderPos + PrevSUCPos
```
- 无加密、无压缩、无校验和
- `_ext.pak` 扩展包优先于基础包加载

## 六、未验证方案

> 有理论依据，值得尝试。

### 6.1 PAK rebuild + 同 pak SMD 替换（最高优先级）

- `repack_pak.py` 存在，rebuild() 未在真 PAK 验证
- 理论成功率 98%（同 pak、dynamic→dynamic、同角色类型）
- 也需要验证 `_ext.pak` 扩展包优先机制是否可用（零风险替代 rebuild）

### 6.2 Hook CharacterMotion 解析器 0x21B46E0

- 修改服务器下发的 Type 值（Static→Dynamic）
- 需要 Apollo CRC 绕过（Frida 写 .text 触发检测）
- **替代方案**：Pymem ReadProcessMemory 读取 CharacterMotion 数据 → 定位 Type 字段 → 堆内存写入（安全）

### 6.3 Hook 工厂函数 0x021C1F00

- 替换创建的对象类型（DStaticActor vtable → DDynamicActor vtable）
- 同样需要 CRC 绕过或堆级别操作

### 6.4 AcquireSMD 入口 0x01EEC130

- 从 SSKF push xref 推导的入口地址
- 从未在 Apollo 移除后测试
- 可用于追踪完整 SMD 加载链

### 6.5 延迟运行时补丁

- 在 Actor 对象完全构造**之后**修改属性（motion type、flags、物理参数）
- 不在构造函数内操作，避免基类差异问题
- 需要进比赛后定位对象实例

### 6.6 NtUserInjectKeyboardInput

- 使用与 NtUserSendInput 不同的 syscall 号
- Apollo 内核层可能未过滤此路径
- 自动登录的潜在突破口
